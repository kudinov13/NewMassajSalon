ПЕРЕЧЕНЬ СОКРАЩЕНИЙ, УСЛОВНЫХ ОБОЗНАЧЕНИЙ И ТЕРМИНОВ

Веб-ресурс – совокупность связанных между собой веб-страниц, объединённых единым доменным именем и предоставляющих пользователям определённый набор функций и информации посредством сети Интернет.

Клиентская часть (frontend) – часть веб-приложения, выполняемая непосредственно в браузере пользователя и отвечающая за отображение пользовательского интерфейса, обработку действий пользователя и взаимодействие с серверной частью посредством программных интерфейсов.

Application Programming Interface (API) – набор правил и протоколов, позволяющий различным программным компонентам обмениваться данными и командами между собой, обеспечивая взаимодействие клиентской и серверной частей веб-приложения.

HyperText Transfer Protocol (HTTP/HTTPS) – протокол прикладного уровня, используемый для передачи гипертекстовых документов и данных между клиентом и сервером в сети Интернет, где HTTPS является его защищённой версией с использованием шифрования.

WebRTC (Web Real-Time Communication) – открытый стандарт и набор технологий, позволяющий организовать передачу аудио- и видеопотоков, а также произвольных данных напрямую между браузерами без необходимости установки дополнительных плагинов или программного обеспечения.

SPA (Single Page Application) – одностраничное приложение, архитектурный подход к построению веб-приложений, при котором взаимодействие с пользователем происходит путём динамической перерисовки текущей страницы без полной её перезагрузки.

REST (Representational State Transfer) – архитектурный стиль взаимодействия компонентов распределённого приложения в сети, основанный на использовании стандартных HTTP-методов для выполнения операций над ресурсами.

UI (User Interface) – пользовательский интерфейс, совокупность визуальных элементов и средств управления, с помощью которых пользователь взаимодействует с программным продуктом.

UX (User Experience) – пользовательский опыт, общее впечатление и совокупность ощущений, которые пользователь получает в процессе взаимодействия с программным продуктом.

Адаптивная вёрстка – подход к созданию веб-страниц, при котором их внешний вид и структура автоматически подстраиваются под различные размеры экранов и типы устройств пользователя.

Cookie – небольшой фрагмент данных, отправляемый сервером и сохраняемый на стороне клиента, используемый для поддержания сессии, аутентификации и хранения пользовательских настроек.

JWT (JSON Web Token) – компактный формат представления данных в виде токена, предназначенный для безопасной передачи информации между сторонами и широко применяемый для аутентификации пользователей в веб-приложениях.


ВВЕДЕНИЕ

Современная индустрия оздоровительных услуг переживает период масштабной цифровой трансформации, обусловленной как развитием интернет-технологий, так и изменением потребительских привычек населения. По данным аналитических исследований, за последние несколько лет спрос на онлайн-запись к специалистам в сфере здоровья и красоты вырос более чем вдвое, а значительная доля клиентов при выборе оздоровительного центра в первую очередь обращает внимание на наличие у него собственного веб-сайта с возможностью дистанционного взаимодействия. Многие небольшие салоны и частные практики по-прежнему ведут приём записей через телефонные звонки или мессенджеры, что создаёт определённые трудности как для самих специалистов, так и для их клиентов.

Отдельного внимания заслуживает ситуация, сложившаяся в Российской Федерации с доступностью средств дистанционной коммуникации. На протяжении последних лет в стране неоднократно вводились ограничения на работу популярных мессенджеров и сервисов видеосвязи, которые прежде активно использовались специалистами для проведения онлайн-консультаций. Блокировки и нестабильная работа привычных каналов связи привели к тому, что значительная часть практикующих психологов, диагностов и консультантов оказалась без надёжного инструмента для удалённого взаимодействия с клиентами. Это обстоятельство существенно повышает актуальность разработки собственных веб-платформ, независимых от сторонних сервисов и способных обеспечить стабильную видеосвязь и проведение прямых трансляций непосредственно в рамках сайта.

Информационный веб-ресурс «Коосмо» представляет собой многофункциональную платформу для оздоровительного центра, объединяющую в себе возможности интернет-магазина, системы онлайн-записи на приём к различным специалистам, проведения видеоконсультаций и прямых трансляций, а также образовательного портала с видеокурсами. Платформа охватывает широкий спектр услуг: от психологических консультаций и комплексной диагностики организма до сеансов звуковой терапии тибетскими чашами и продажи специализированных товаров для самомассажа, ароматерапии и здорового питания. Особенностью проекта является встроенная система видеокомнат на основе технологии WebRTC, позволяющая специалистам и клиентам взаимодействовать в режиме реального времени без использования сторонних приложений, а также модуль прямых трансляций, дающий возможность проводить образовательные и информационные эфиры с неограниченным числом зрителей.

Разработка и создание веб-ресурса выполняется в рамках группового студенческого проекта с делением ролей и соответствующих им задач по следующим равнозначным для исполнения частям: дизайн пользовательского интерфейса (UI/UX), клиентская часть (frontend), серверная часть (backend). Целью данной выпускной квалификационной работы является разработка клиентской части информационного веб-ресурса «Коосмо» с реализацией следующих функциональных модулей: главная страница с представлением услуг, система регистрации и авторизации пользователей, интернет-магазин с корзиной и оформлением заказов, онлайн-запись к психологу с видеокомнатой, онлайн-запись на диагностику с видеокомнатой, запись на сеансы тибетских чаш, модуль прямых трансляций с чатом, каталог видеокурсов с возможностью покупки и просмотра, личный кабинет пользователя, панели управления для специалистов и администратора, а также адаптивная вёрстка для корректного отображения на устройствах с различными размерами экранов.


1 ПОСТАНОВКА ЗАДАЧИ ВКР

1.1 Бизнес-требования

Под бизнес-требованиями в контексте данной работы понимается совокупность стратегических целей и задач, которые заказчик планирует достичь путём внедрения и эксплуатации разрабатываемого веб-ресурса. Оздоровительный центр «Коосмо» до настоящего момента не располагал собственным полноценным веб-присутствием и привлекал клиентов главным образом через устные рекомендации, социальные сети и сторонние площадки-агрегаторы, что существенно ограничивало охват целевой аудитории и не позволяло выстроить прямой канал коммуникации с потребителями услуг.

Первостепенной бизнес-задачей разрабатываемой платформы является увеличение клиентского потока и повышение конверсии посетителей в постоянных клиентов центра. Собственный веб-сайт с профессиональным дизайном и удобной навигацией формирует доверие к бренду, повышает его узнаваемость среди потенциальных клиентов и служит круглосуточной витриной предоставляемых услуг. Наличие системы онлайн-записи устраняет барьер первого контакта, позволяя клиенту самостоятельно выбрать удобное время визита без необходимости совершать телефонный звонок, что особенно важно для аудитории, предпочитающей цифровые каналы взаимодействия.

Не менее значимым бизнес-требованием является расширение географии предоставления услуг за счёт внедрения механизма дистанционных консультаций. Встроенная система видеосвязи позволяет психологу и диагносту центра проводить полноценные приёмы с клиентами, находящимися в других городах и регионах, что многократно увеличивает потенциальную клиентскую базу и снимает географические ограничения. Модуль прямых трансляций решает задачу масштабного информационного охвата: образовательные эфиры привлекают новую аудиторию, формируют экспертный образ специалистов центра и стимулируют интерес к основным услугам.

Интернет-магазин, встроенный в платформу, открывает дополнительный канал монетизации, позволяя реализовывать товары для ароматерапии, SPA-рецепты, биологически активные добавки и курсы самомассажа непосредственно на сайте. Каталог видеокурсов с возможностью покупки и просмотра создаёт пассивный источник дохода для специалистов центра и одновременно выполняет образовательную функцию, повышая лояльность клиентов. Совокупность перечисленных коммерческих инструментов направлена на рост среднего чека и увеличение общей выручки оздоровительного центра.

С точки зрения маркетинга и продвижения, наличие полноценного сайта с разнообразным контентом положительно сказывается на позициях в поисковой выдаче, органически привлекая новых посетителей без дополнительных затрат на рекламу. Каждая страница услуги, каждый анонс трансляции и каждый товар в каталоге представляют собой точку входа для потенциального клиента, что формирует устойчивый поток органического трафика. Система регистрации и ведения профилей пользователей создаёт базу контактных данных, которую администратор может использовать для адресных уведомлений о новых услугах, акциях и мероприятиях, что является эффективным инструментом удержания и повторного вовлечения клиентов.

Таким образом, разрабатываемый веб-ресурс призван решить комплекс бизнес-задач: привлечение нового клиентского трафика через органическое продвижение в поисковых системах, повышение конверсии за счёт удобной системы онлайн-записи, расширение географии услуг благодаря видеоконсультациям, рост продаж через встроенный интернет-магазин и каталог курсов, а также укрепление бренда и экспертного имиджа оздоровительного центра посредством прямых трансляций и образовательного контента.

1.2 Пользовательские требования

Ознакомившись с потребностями заказчика и проанализировав целевую аудиторию веб-ресурса, были сформулированы пользовательские требования, которые описывают функциональные возможности системы с точки зрения конечного пользователя и определяют, каким образом он должен взаимодействовать с веб-приложением для решения своих задач.

Незарегистрированный пользователь, впервые попавший на сайт, должен иметь возможность ознакомиться с полным перечнем предоставляемых центром услуг, просмотреть каталог товаров интернет-магазина, изучить расписание предстоящих прямых трансляций, а также ознакомиться с информационными материалами и руководством пользователя. При этом для совершения записи на приём, покупки товаров или курсов, просмотра приобретённых видеокурсов и участия в трансляциях пользователь должен пройти процедуру регистрации, указав свои персональные данные, включая ФИО, номер телефона и адрес электронной почты, после чего авторизоваться в системе.

Зарегистрированный пользователь должен иметь доступ к личному кабинету, где он может просматривать и редактировать свои персональные данные, а также видеть историю записей к специалистам и совершённых покупок. Система онлайн-записи должна предоставлять пользователю возможность выбрать интересующую услугу, ознакомиться с доступными временными слотами в расписании специалиста, выбрать подходящее время и подтвердить запись. При оформлении записи форма должна автоматически заполняться данными из профиля пользователя, однако должна сохраняться возможность указать другой номер телефона или иные контактные данные в случае необходимости. Для услуг, предполагающих дистанционное взаимодействие, таких как психологическая консультация или диагностика, пользователь должен иметь возможность войти в видеокомнату непосредственно через интерфейс веб-приложения в назначенное время приёма. Кроме того, пользователь должен иметь возможность добавлять товары в корзину, управлять её содержимым и оформлять заказы, просматривать и приобретать видеокурсы, подключаться к запланированным прямым трансляциям и участвовать в чате во время их проведения.

1.3 Системные требования

Разработка клиентской части веб-ресурса «Коосмо» предполагает создание одностраничного приложения, функционирующего в среде современных веб-браузеров и взаимодействующего с серверной частью посредством программного интерфейса по протоколу HTTP. Системные требования к клиентской части определяют ключевые параметры, которые должны быть обеспечены для стабильной и эффективной работы приложения.

Клиентская часть должна быть реализована по принципу одностраничного приложения, при котором навигация между разделами осуществляется без полной перезагрузки страницы, что обеспечивает плавное и быстрое взаимодействие пользователя с интерфейсом. Обмен данными между клиентом и сервером должен происходить в формате JSON с использованием стандартных HTTP-методов в соответствии с принципами REST-архитектуры. Аутентификация пользователя должна поддерживаться на основе токенов, сохраняемых в cookie-файлах с флагом httpOnly для обеспечения защиты от атак типа XSS.

Модуль видеосвязи должен быть реализован с использованием технологии WebRTC, обеспечивающей установление прямого однорангового соединения между участниками консультации без передачи медиапотоков через сервер. Модуль прямых трансляций должен поддерживать передачу видеопотока от администратора неограниченному числу зрителей, а также обеспечивать работу текстового чата в реальном времени. Веб-приложение должно корректно функционировать в актуальных версиях браузеров Google Chrome, Mozilla Firefox, Microsoft Edge и Safari. Структура кода клиентской части должна быть модульной и обеспечивать возможность расширения функциональности путём добавления новых компонентов и страниц без необходимости переработки существующей кодовой базы.

1.4 Требования к пользовательскому интерфейсу

Графический пользовательский интерфейс разрабатываемого веб-ресурса является основным средством взаимодействия посетителя с платформой и во многом определяет его первое впечатление о бренде оздоровительного центра, готовность продолжить изучение предоставляемых услуг и совершить целевое действие. Требования к интерфейсу затрагивают как визуальную составляющую, так и функциональные аспекты организации экранных форм.

Дизайн интерфейса должен быть выдержан в единой стилистике, отражающей тематику оздоровительных и wellness-услуг, с использованием мягкой тёплой цветовой палитры, скруглённых элементов управления и сдержанной типографики, создающей ощущение спокойствия и доверия. Навигация по разделам сайта должна быть простой и интуитивно понятной, позволяя пользователю в минимальное число кликов добраться до интересующей его информации или функции. Интерфейс должен обеспечивать визуальную обратную связь при выполнении пользовательских действий: загрузочные индикаторы при выполнении запросов, сообщения об успешном завершении операций или допущенных ошибках, анимации переходов между страницами.

Особое внимание должно быть уделено адаптивности интерфейса, то есть его способности корректно отображаться и сохранять полную функциональность на устройствах с различными размерами экранов. На мобильных устройствах и планшетах основное навигационное меню должно трансформироваться в компактное «бургер-меню» с плавной анимацией открытия и закрытия. Все интерактивные элементы, включая кнопки, поля ввода и ссылки, должны иметь достаточные размеры области нажатия для комфортной работы на сенсорных экранах. Таблицы с данными на административных панелях должны поддерживать горизонтальную прокрутку на узких экранах, не нарушая общую компоновку страницы.

2.1 Описание предметной области задачи ВКР

Точное и всестороннее описание предметной области является необходимым условием для проектирования программного средства, отвечающего реальным потребностям его будущих пользователей. Понимание специфики бизнес-процессов оздоровительного центра, характера предоставляемых услуг и способов взаимодействия специалистов с клиентами позволяет сформулировать обоснованные требования к функциональности и архитектуре клиентской части веб-приложения.

Предметная область разрабатываемого веб-ресурса охватывает сферу оздоровительных и wellness-услуг, включающих психологическое консультирование, комплексную диагностику состояния организма по внешним признакам, звуко-вибрационную терапию с использованием тибетских поющих чаш, а также розничную реализацию специализированных товаров и образовательных материалов в сфере самомассажа, ароматерапии, правильного питания и биологически активных добавок. Каждое из перечисленных направлений обладает собственной спецификой в части организации записи, проведения приёма и коммуникации между специалистом и клиентом, что предъявляет различные требования к функциональным модулям клиентской части.

Психологическое консультирование и диагностика организма предполагают дистанционный формат проведения с использованием средств видеосвязи, что требует реализации модуля видеокомнат с поддержкой двусторонней аудио- и видеосвязи в реальном времени. Звуко-вибрационная терапия тибетскими чашами, напротив, проводится исключительно в очном формате, что определяет необходимость иного подхода к организации записи: специалисту не требуется функциональность видеосвязи, однако необходима возможность просматривать контактные данные записавшихся клиентов для телефонного согласования деталей приёма. Розничное направление предполагает наличие каталога товаров с возможностью добавления в корзину и оформления заказа, а образовательное направление требует реализации каталога видеокурсов с механизмом покупки и предоставления доступа к учебным материалам.

2.1.1 Информационные объекты предметной области и взаимосвязи между ними

Корректное выделение информационных объектов предметной области и установление взаимосвязей между ними является фундаментом проектирования любой информационной системы. Идентификация ключевых сущностей, с которыми оперирует система, и описание характера их отношений позволяют определить структуру пользовательского интерфейса, состав экранных форм и логику переходов между ними.

В рамках разрабатываемого веб-ресурса можно выделить следующие основные информационные объекты:

1 Пользователь — центральный объект системы, характеризуемый учётными данными (логин, пароль), персональной информацией (ФИО, номер телефона, адрес электронной почты) и ролью в системе (обычный пользователь, психолог, специалист по тибетским чашам, администратор). Пользователь является связующим звеном между остальными объектами системы, поскольку выступает субъектом записей на приём, покупок товаров и курсов, участия в трансляциях.

2 Услуга — информационный объект, представляющий конкретное направление деятельности оздоровительного центра: психологическая консультация, диагностика по одному из пяти типов (ногти, язык, глаза, кожа, тело и осанка), сеанс тибетских чаш. Каждая услуга характеризуется описанием, стоимостью и форматом проведения (очный или дистанционный).

3 Слот расписания — объект, определяющий конкретную дату и время, доступные для записи к определённому специалисту. Слот может находиться в одном из двух состояний: свободный или забронированный. Для сеансов тибетских чаш слот дополнительно характеризуется городом проведения.

4 Запись на приём — объект, фиксирующий факт бронирования конкретного временного слота конкретным пользователем. Запись содержит контактные данные клиента (ФИО, телефон, электронная почта), ссылку на слот расписания и статус (забронирована, идёт приём, завершена, отменена). Для дистанционных услуг запись дополнительно связана с объектом видеокомнаты.

5 Товар — объект каталога интернет-магазина, характеризуемый наименованием, описанием, ценой, категорией (ароматерапия, SPA-рецепты, курсы самомассажа, рецепты питания, БАДы) и изображением. Товар может быть связан с видеокурсом, если представляет собой цифровой продукт.

6 Корзина и заказ — корзина представляет собой временную коллекцию товаров, выбранных пользователем для покупки, с указанием количества каждого товара. При оформлении заказа содержимое корзины фиксируется в виде заказа с присвоением статуса и сохранением в истории покупок.

7 Видеокурс — объект, содержащий набор видеоуроков, объединённых общей тематикой. Курс характеризуется названием, описанием, стоимостью и изображением-обложкой. Доступ к просмотру видеоуроков курса предоставляется после его покупки.

8 Трансляция — объект, описывающий запланированный или проводимый прямой эфир. Характеризуется заголовком, описанием, датой и временем проведения, именем ведущего, стоимостью участия и статусом (запланирована, идёт, завершена). Связан с объектом комнаты трансляции, обеспечивающей передачу видеопотока и работу чата.

9 Видеокомната — объект, обеспечивающий двустороннюю видеосвязь между специалистом и клиентом в рамках дистанционной консультации. Связан с конкретной записью на приём.

2.1.2 Информационные потребности пользователей

Определение информационных потребностей пользователей является ключевым шагом при проектировании пользовательского интерфейса, поскольку позволяет установить, какие именно данные система должна предоставлять каждой категории пользователей и в каком виде эта информация должна быть представлена для наиболее эффективного решения их задач.

Разрабатываемый веб-ресурс призван удовлетворить следующие информационные потребности:

1 Потенциальный клиент нуждается в исчерпывающей информации о предоставляемых услугах — описании каждого направления, квалификации специалистов, стоимости и формате проведения сеансов. Кроме того, ему необходим удобный доступ к актуальному расписанию с возможностью выбора свободного временного слота и оперативного оформления записи.

2 Постоянный клиент заинтересован в быстром доступе к личному кабинету, где сосредоточена информация о предстоящих и прошедших записях, приобретённых курсах и истории покупок. Ему необходима возможность оперативно перейти к видеокомнате для дистанционной консультации или к просмотру купленного видеокурса.

3 Специалист (психолог, диагност) нуждается в информации о записавшихся клиентах: ФИО, контактный телефон, электронная почта, дата и время записи, а также в инструментах управления расписанием и проведения видеоконсультаций.

4 Специалист по тибетским чашам нуждается в аналогичной контактной информации о клиентах, однако вместо функциональности видеосвязи ему требуются удобные средства для телефонного контакта с клиентом с целью согласования деталей очного приёма.

5 Администратор нуждается в сводной статистике по активности платформы, полном списке зарегистрированных пользователей с их контактными данными, инструментах управления контентом (товары, курсы, трансляции, руководство пользователя) и возможности назначения ролей пользователям.

2.1.3 Методы работы с информационными объектами предметной области

Описание методов работы с информационными объектами позволяет формализовать способы представления, обработки и передачи данных в рамках клиентской части веб-приложения. В данном подразделе рассматриваются как способы визуального отображения информации пользователю, так и технологические механизмы её получения и передачи.

2.1.3.1 Способы хранения информации на стороне клиента

Клиентская часть веб-приложения, в отличие от серверной, не взаимодействует напрямую с реляционной базой данных, однако нуждается в собственных механизмах хранения информации для обеспечения корректной работы интерфейса, поддержания состояния пользовательских сеансов и сохранения пользовательских предпочтений между визитами.

Основным механизмом хранения оперативных данных в клиентской части является внутреннее состояние React-компонентов, управляемое посредством хука useState. Каждый страничный компонент хранит в своём состоянии данные, загруженные с сервера (списки товаров, слоты расписания, записи на приём, информацию о пользователе), значения полей форм, флаги загрузки и сообщения об ошибках. Данный подход обеспечивает мгновенную реактивность интерфейса: любое изменение состояния автоматически вызывает перерисовку соответствующих элементов на странице.

Для хранения данных между сеансами работы пользователя используется механизм локального хранилища браузера (localStorage). В частности, согласие пользователя на обработку персональных данных фиксируется в localStorage и проверяется при каждом последующем посещении сайта, что позволяет не отображать уведомление повторно. Аутентификационный токен хранится в cookie-файле с флагом httpOnly, что исключает доступ к нему из JavaScript-кода и обеспечивает автоматическую отправку при каждом запросе к серверу.

Содержимое корзины интернет-магазина привязано к серверному хранилищу через API и синхронизируется при каждом действии пользователя (добавление, изменение количества, удаление товара), что гарантирует сохранность данных корзины при обновлении страницы или смене устройства.

2.1.3.2 Математические модели, используемые для обработки информации

В контексте клиентской части веб-приложения математические модели применяются для обработки и преобразования данных непосредственно в браузере пользователя, что позволяет обеспечить мгновенную обратную связь без обращения к серверу и повысить отзывчивость интерфейса.

Валидация пользовательского ввода основана на проверке соответствия введённых данных заданным ограничениям. При регистрации проверяется заполненность обязательных полей (ФИО, телефон, логин, пароль), минимальная длина пароля и совпадение пароля с его подтверждением. Данная модель реализована как последовательность условных проверок, выполняемых до отправки данных на сервер, что позволяет исключить заведомо некорректные запросы.

Вычисление итоговой стоимости корзины выполняется на стороне клиента по формуле: S = Σ(Pi × Qi), где S — итоговая сумма заказа, Pi — цена i-го товара, Qi — количество единиц i-го товара в корзине. Данное вычисление производится при каждом изменении содержимого корзины и немедленно отображается пользователю без обращения к серверу.

Форматирование дат и времени для отображения расписания специалистов выполняется посредством преобразования серверного формата даты (строка ISO 8601) в локальный формат, привычный для русскоязычного пользователя (ДД.ММ.ГГГГ, ЧЧ:ММ). Алгоритм определения доступности временного слота основан на сравнении текущего времени с временем начала слота: слоты, время которых уже наступило, автоматически помечаются как недоступные для записи.

Правило «2 часа до приёма» определяет порог, за которым пользователь получает возможность войти в видеокомнату: кнопка входа отображается только если до начала записи осталось менее двух часов. Данная логика реализована как сравнение временных меток текущего момента и времени начала консультации.

2.1.3.3 Способы интерпретации и визуального представления информации

Эффективное визуальное представление информации является одним из определяющих факторов удобства использования программного средства. Грамотно организованная подача данных позволяет пользователю быстро находить нужную информацию, интуитивно понимать доступные действия и уверенно совершать целевые операции.

В клиентской части веб-ресурса «Коосмо» информация представляется пользователю посредством нескольких визуальных форматов. Каталог услуг на главной странице реализован в виде карточного интерфейса, где каждая карточка содержит изображение, краткое описание услуги и навигационную стрелку для перехода к детальной информации. Подобный подход позволяет пользователю за несколько секунд охватить весь спектр предложений центра и выбрать интересующее направление.

Каталог товаров интернет-магазина представлен в виде сетки карточек с фильтрацией по категориям, что обеспечивает быструю навигацию по ассортименту. Расписание специалистов отображается в табличном формате с группировкой по датам и визуальной индикацией доступности временных слотов. Прямые трансляции представлены в виде списка с выделением текущих эфиров и отображением информации о дате, времени, ведущем и стоимости участия. Административные панели используют табличный формат для отображения структурированных данных о пользователях, записях и заказах с возможностью поиска и фильтрации.

2.1.3.4 Технологии получения и передачи информации

Выбор и обоснование технологий получения и передачи информации определяют архитектурные решения, лежащие в основе взаимодействия клиентской части с серверной и с пользовательским окружением. Корректная организация этих процессов обеспечивает отзывчивость интерфейса, целостность данных и безопасность пользовательских сессий.

Взаимодействие клиентской части с серверной осуществляется посредством HTTP-запросов к REST API. При загрузке каждого раздела приложение направляет запрос к соответствующему серверному маршруту и получает данные в формате JSON, после чего выполняет их отображение на странице. Все запросы, требующие идентификации пользователя, сопровождаются автоматической отправкой cookie-файла с токеном аутентификации, что обеспечивается параметром credentials: "include" в конфигурации запросов.

При проведении видеоконсультаций используется технология WebRTC, обеспечивающая установление прямого однорангового соединения между браузерами участников. Сигнальный обмен для согласования параметров соединения осуществляется через серверную часть посредством механизма поллинга: клиентское приложение периодически запрашивает у сервера наличие новых сигнальных сообщений от удалённой стороны. Для обхода сетевых ограничений используются STUN-серверы, обеспечивающие определение внешних адресов участников.

Прямые трансляции также построены на базе WebRTC, однако в данном случае ведущий выступает единственным источником медиапотока, а каждый зритель устанавливает с ним независимое одноранговое соединение. Параллельно с видеопотоком функционирует текстовый чат, сообщения которого передаются через серверную часть и обновляются на стороне клиента с помощью периодического опроса.

Загрузка файлов, таких как изображения товаров, видеозаписи курсов и превью трансляций, осуществляется посредством отправки данных в формате multipart/form-data с использованием объекта FormData, что позволяет передавать бинарные данные совместно с текстовыми полями в одном запросе.

2.1.5 Концептуальное обоснование разработки

Концептуальное обоснование разработки представляет собой аргументированное объяснение необходимости создания нового программного продукта с учётом выявленных потребностей заказчика, особенностей предметной области и результатов анализа существующих решений.

Разработка собственной клиентской части информационного веб-ресурса «Коосмо» обоснована следующими аргументами. Во-первых, ни одно из рассмотренных существующих решений не предоставляет комплексной функциональности, объединяющей в себе систему онлайн-записи, видеоконсультации, прямые трансляции, интернет-магазин и образовательную платформу. Заказчику пришлось бы использовать несколько разрозненных сервисов с различными интерфейсами, что неудобно как для специалистов, так и для клиентов, а совокупная стоимость подписок существенно превысила бы затраты на собственную разработку. Во-вторых, нестабильность работы сторонних средств видеосвязи на территории Российской Федерации делает критически важным наличие встроенного модуля видеокомнат, функционирующего непосредственно в рамках собственного веб-ресурса и не зависящего от политики блокировок. В-третьих, собственная платформа обеспечивает полный контроль над пользовательскими данными, дизайном, функциональностью и дальнейшим развитием продукта, что невозможно при использовании сторонних облачных решений.

2.3 Функциональные требования

Формулирование функциональных требований является одним из наиболее ответственных этапов проектирования программного средства, поскольку именно они определяют полный набор операций, которые система должна выполнять для удовлетворения потребностей каждой категории пользователей. Функциональные требования описывают поведение системы в ответ на действия пользователя и служат основой для последующей разработки и тестирования.

Клиентская часть веб-ресурса «Коосмо» должна обеспечивать реализацию следующих функциональных возможностей:

1 Регистрация пользователя с указанием логина, пароля, ФИО, номера телефона и адреса электронной почты, с валидацией обязательных полей и проверкой совпадения паролей на стороне клиента.

2 Авторизация пользователя по логину и паролю с автоматическим обновлением состояния интерфейса после успешного входа, отображением элементов управления, доступных авторизованному пользователю.

3 Выход из системы с очисткой сессии и перенаправлением на главную страницу.

4 Отображение и редактирование профиля пользователя с возможностью изменения ФИО, номера телефона и адреса электронной почты.

5 Просмотр каталога товаров интернет-магазина с фильтрацией по категориям, добавление товаров в корзину, управление количеством товаров в корзине, удаление товаров из корзины и оформление заказа.

6 Просмотр истории покупок с детальной информацией о каждом заказе.

7 Онлайн-запись на психологическую консультацию: просмотр доступных слотов, выбор времени, автоматическое заполнение формы данными из профиля с возможностью ручного редактирования, подтверждение записи.

8 Онлайн-запись на диагностику организма: аналогичная функциональность с учётом специфики данного вида услуг.

9 Онлайн-запись на сеанс тибетских чаш: просмотр доступных слотов с фильтрацией по городу, выбор времени, заполнение контактных данных, подтверждение записи.

10 Двусторонняя видеосвязь в рамках психологической консультации и диагностики: вход в видеокомнату, управление камерой и микрофоном, демонстрация экрана, завершение сеанса.

11 Просмотр списка прямых трансляций, подключение к текущей трансляции в качестве зрителя, участие в текстовом чате во время трансляции.

12 Проведение прямой трансляции администратором: запуск эфира, передача видеопотока зрителям, управление чатом.

13 Просмотр каталога видеокурсов, покупка курса, просмотр приобретённых видеоуроков.

14 Панель психолога и диагноста: управление расписанием (добавление и удаление слотов), просмотр записей с контактными данными клиентов, управление видеокомнатами.

15 Панель специалиста по тибетским чашам: управление расписанием по городам, просмотр записей с контактными данными клиентов (ФИО, телефон, электронная почта) с возможностью перехода к звонку.

16 Административная панель: просмотр статистики, управление пользователями и их ролями, управление товарами и курсами, управление трансляциями и лабораториями, просмотр журнала активности.

17 Отображение всплывающего уведомления о согласии на обработку персональных данных при первом посещении сайта.

18 Адаптивное отображение интерфейса на устройствах с различными размерами экранов, включая мобильное бургер-меню.

2.3.1 Определение функциональных возможностей

Определение функциональных возможностей предполагает систематизацию ранее перечисленных требований в виде структурированных функциональных блоков, каждый из которых отвечает за определённую область функциональности приложения. Подобная группировка позволяет чётко распределить ответственность между модулями системы и обеспечить их независимую разработку и тестирование.

Функциональные возможности клиентской части веб-ресурса «Коосмо» можно сгруппировать в следующие блоки:

1 Блок аутентификации — регистрация, авторизация, выход из системы, управление сессией.

2 Блок профиля пользователя — отображение и редактирование персональных данных, просмотр записей и истории покупок.

3 Блок интернет-магазина — каталог товаров, корзина, оформление заказа, история заказов.

4 Блок записи на приём — расписание специалистов, форма записи, автозаполнение данных, просмотр своих записей.

2.3.2 Описание прецедентов

Описание прецедентов представляет собой детализированные сценарии взаимодействия пользователей с системой, позволяющие конкретизировать ожидаемое поведение приложения в типичных ситуациях использования и служащие основой для разработки тестовых сценариев.

Прецедент 1 — Регистрация нового пользователя. Актор: незарегистрированный пользователь. Пользователь открывает страницу регистрации, заполняет поля ФИО, телефон, электронная почта, логин, пароль, подтверждение пароля. Система проверяет заполненность обязательных полей и совпадение паролей. При успешной валидации данные отправляются на сервер. В случае успеха отображается сообщение об успешной регистрации с предложением авторизоваться. В случае ошибки отображается соответствующее сообщение.

Прецедент 2 — Запись на психологическую консультацию. Актор: зарегистрированный пользователь. Пользователь переходит на страницу записи к психологу. Система загружает и отображает доступные временные слоты. Пользователь выбирает дату и время. Система открывает форму подтверждения, автоматически заполненную данными из профиля. Пользователь при необходимости корректирует данные и подтверждает запись. Система отправляет запрос на сервер, при успехе отображает подтверждение и перенаправляет в личный кабинет.

Прецедент 3 — Проведение видеоконсультации. Актор: психолог, зарегистрированный пользователь (клиент). Специалист входит в панель управления и нажимает кнопку входа в видеокомнату для соответствующей записи. Система запрашивает доступ к камере и микрофону, устанавливает WebRTC-соединение. Клиент аналогично входит в комнату через свой личный кабинет. Оба участника видят и слышат друг друга. По завершении специалист закрывает комнату, запись получает статус «Завершена».

Прецедент 4 — Покупка товара через интернет-магазин. Актор: зарегистрированный пользователь. Пользователь переходит в раздел «Магазин», просматривает товары, нажимает «В корзину» на выбранном товаре. Система добавляет товар в корзину и отображает обновлённый счётчик. Пользователь переходит в корзину, проверяет содержимое, нажимает «Оформить заказ». Система фиксирует заказ и очищает корзину.

2.4 Нефункциональные требования

Нефункциональные требования определяют качественные характеристики программного продукта, которые не связаны напрямую с конкретными функциями, но оказывают существенное влияние на пользовательский опыт, надёжность и удобство сопровождения системы. Соблюдение нефункциональных требований обеспечивает конкурентоспособность продукта и удовлетворённость пользователей.

Требования к производительности определяют допустимое время отклика интерфейса на действия пользователя. Переключение между разделами приложения должно происходить без видимых задержек, поскольку навигация реализована на стороне клиента без полной перезагрузки страницы. Загрузка данных с сервера при открытии раздела не должна превышать двух-трёх секунд при стандартном интернет-соединении. Установление видеосвязи через WebRTC должно занимать не более пяти секунд после разрешения доступа к камере и микрофону.

Требования к надёжности обязывают клиентскую часть корректно обрабатывать ошибочные ситуации: отсутствие сетевого соединения, ошибки сервера, попытки доступа к защищённым разделам без авторизации. В каждом из перечисленных случаев пользователю должно быть отображено информативное сообщение с описанием проблемы и рекомендациями к дальнейшим действиям. Потеря соединения во время видеоконсультации не должна приводить к потере данных записи, а при восстановлении связи пользователь должен иметь возможность повторно войти в комнату.

Требования к безопасности предписывают реализацию механизмов защиты пользовательских данных на стороне клиента. Пароли не должны передаваться в открытом виде в URL-адресах или сохраняться в локальном хранилище браузера. Токен аутентификации должен храниться в cookie-файле с флагом httpOnly, исключающим его доступность из JavaScript-кода, что предотвращает кражу сессии при потенциальных XSS-атаках. При первом посещении сайта пользователь должен быть уведомлён о сборе и обработке персональных данных посредством всплывающего уведомления с возможностью подтверждения согласия.

Требования к удобству использования определяют, что интерфейс должен быть интуитивно понятным для пользователей с различным уровнем компьютерной грамотности. Основные пользовательские сценарии, такие как запись на приём или покупка товара, должны реализовываться в минимальное количество шагов. Формы ввода должны содержать подсказки-заполнители, валидацию в реальном времени и понятные сообщения об ошибках. Интерфейс должен быть полностью функционален на устройствах с размером экрана от 320 пикселей в ширину.

Требования к сопровождаемости предусматривают модульную организацию кодовой базы, при которой каждый функциональный блок реализован в виде независимого компонента с чётко определённой зоной ответственности. Компонентный подход обеспечивает возможность внесения изменений в один модуль без риска повреждения логики работы остальных частей приложения, что существенно упрощает дальнейшее сопровождение и развитие продукта.


3 ВЫБОР ПРОГРАММНЫХ СРЕД И СРЕДСТВ РАЗРАБОТКИ

3.1 Сравнительный анализ имеющихся возможностей по выбору средств разработки

Выбор технологического стека для разработки клиентской части веб-приложения является ответственным решением, определяющим не только скорость и удобство процесса разработки, но и производительность, масштабируемость и сопровождаемость итогового продукта на протяжении всего его жизненного цикла. Современный рынок инструментов фронтенд-разработки предлагает широкий спектр библиотек и фреймворков, каждый из которых обладает собственными преимуществами и ограничениями, что делает необходимым проведение их сравнительного анализа применительно к задачам конкретного проекта.

В качестве основного языка программирования для клиентской части рассматривались JavaScript и TypeScript. JavaScript является стандартным языком веб-браузеров и обладает наибольшей экосистемой библиотек, однако его динамическая типизация затрудняет обнаружение ошибок на этапе разработки и усложняет рефакторинг кода по мере роста проекта. TypeScript представляет собой надмножество JavaScript, дополняющее его статической системой типов, что позволяет выявлять значительную часть ошибок ещё до запуска приложения, а также обеспечивает более качественную поддержку со стороны редакторов кода в виде автодополнения и подсказок. С учётом масштаба разрабатываемого проекта и необходимости поддержки множества модулей с различными типами данных выбор был сделан в пользу TypeScript.

Для построения пользовательского интерфейса были рассмотрены три наиболее популярных решения: React, Vue.js и Angular. Angular является полноценным фреймворком с жёстко определённой архитектурой, встроенной системой маршрутизации и управления формами, что делает его удобным для крупных корпоративных приложений, однако порог вхождения в Angular существенно выше, а объём генерируемого кода больше по сравнению с альтернативами. Vue.js отличается простотой освоения и лаконичным синтаксисом, однако его экосистема несколько уступает по масштабу экосистеме React, а количество готовых решений для специфических задач, таких как интеграция с WebRTC, заметно меньше. React представляет собой библиотеку для построения пользовательских интерфейсов на основе компонентного подхода, обладающую крупнейшей экосистемой, обширной документацией и широким сообществом разработчиков. Компонентная модель React с использованием хуков обеспечивает гибкость и переиспользуемость кода, а декларативный подход к описанию интерфейса упрощает понимание и сопровождение кодовой базы. По совокупности факторов для реализации клиентской части был выбран React.

Для стилизации компонентов рассматривались три подхода: традиционные CSS-модули, препроцессор SASS и утилитарный CSS-фреймворк Tailwind CSS. CSS-модули обеспечивают изоляцию стилей на уровне компонента, однако требуют создания отдельных файлов стилей и увеличивают число файлов в проекте. SASS расширяет возможности CSS переменными, миксинами и вложенностью, но не решает проблему именования классов и не ускоряет процесс вёрстки принципиально. Tailwind CSS реализует утилитарный подход, при котором стилизация осуществляется непосредственно в разметке компонента посредством предопределённых служебных классов, что значительно ускоряет разработку, устраняет необходимость в придумывании имён классов и обеспечивает единообразие визуального стиля. С учётом необходимости создания адаптивного интерфейса и большого количества страниц был выбран Tailwind CSS.

Для организации маршрутизации в одностраничном приложении была выбрана библиотека React Router DOM, являющаяся стандартным решением для навигации в React-приложениях и обеспечивающая декларативное описание маршрутов с поддержкой вложенной маршрутизации и динамических параметров.

В разрабатываемом приложении определено более двадцати маршрутов, включая вложенные маршруты для различных типов диагностики и параметризованные маршруты для видеокомнат, трансляций и курсов.

Tailwind CSS версии 3.4 — утилитарный CSS-фреймворк, предоставляющий набор низкоуровневых служебных классов для стилизации элементов непосредственно в HTML-разметке. Tailwind CSS поддерживает адаптивные модификаторы (sm, md, lg, xl), позволяющие задавать различные стили для разных размеров экранов, псевдоклассы состояний (hover, focus, active) и произвольные значения в квадратных скобках для нестандартных случаев. Встроенный механизм очистки неиспользуемых классов обеспечивает минимальный размер итогового CSS-файла.

Create React App версии 5.0 — инструмент инициализации и сборки React-приложений, предоставляющий преднастроенную конфигурацию Webpack, Babel и средств разработки. CRA обеспечивает горячую перезагрузку модулей в процессе разработки, поддержку TypeScript без дополнительной настройки, проксирование API-запросов на серверную часть и оптимизированную сборку для производственного окружения.

WebRTC — открытая технология для организации аудио- и видеосвязи непосредственно между браузерами. Fetch API — встроенный в браузер интерфейс для выполнения HTTP-запросов. В данном проекте Fetch API используется для всех взаимодействий с серверной частью: получение и отправка данных в формате JSON, загрузка файлов через FormData, управление аутентификационными cookie-файлами посредством параметра credentials.

Git — распределённая система управления версиями, используемая для отслеживания изменений в кодовой базе, организации совместной работы участников команды и хранения истории разработки в удалённом репозитории.

Visual Studio Code — редактор исходного кода, используемый в качестве основной среды разработки. Обеспечивает подсветку синтаксиса TypeScript и JSX, автодополнение на основе типов, встроенный терминал и обширную библиотеку расширений для повышения продуктивности разработчика.

4 АЛГОРИТМЫ РЕШЕНИЯ ПОСТАВЛЕННОЙ ЗАДАЧИ

4.1 Этапы реализации

Описание этапов реализации позволяет формализовать последовательность шагов, которые были выполнены для перехода от сформулированных требований к работающему программному продукту. Структурирование процесса разработки на отдельные этапы обеспечивает контроль над ходом работ, позволяет своевременно выявлять отклонения от плана и вносить необходимые коррективы.

Разработка клиентской части веб-ресурса «Коосмо» была выполнена в несколько последовательных этапов.

На первом этапе была проведена инициализация проекта с использованием Create React App с шаблоном TypeScript. Была настроена структура директорий: каталог components для переиспользуемых компонентов, каталог views для страничных компонентов и каталог services для модулей взаимодействия с серверным API. Был подключён и настроен Tailwind CSS, определена цветовая палитра и типографические параметры, соответствующие дизайн-макету.

На втором этапе была реализована система маршрутизации с использованием React Router DOM. Был создан корневой компонент Layout, обеспечивающий общую обёртку для всех страниц, и определены маршруты для каждого раздела приложения. На этом же этапе был реализован компонент PageTransition, обеспечивающий плавную анимацию при переключении между страницами посредством управления прозрачностью элементов.

На третьем этапе были реализованы модули аутентификации: страницы регистрации и авторизации с формами ввода, клиентской валидацией и обработкой серверных ответов. Был создан сервисный модуль API, инкапсулирующий все HTTP-запросы к серверной части и обеспечивающий единообразную обработку ошибок. Также был реализован компонент Header с навигационным меню, отображением статуса авторизации и адаптивным бургер-меню для мобильных устройств.

На четвёртом этапе были реализованы основные бизнес-модули: главная страница с представлением услуг, каталог интернет-магазина с фильтрацией по категориям и управлением корзиной, страницы описания услуг (психология, диагностика, тибетские чаши, анализы), страницы записи на приём к каждому типу специалиста с выбором слотов из расписания и автозаполнением формы данными профиля.

На пятом этапе были реализованы модули видеосвязи и трансляций. Видеокомната реализована как отдельный страничный компонент, управляющий локальным и удалённым медиапотоками через API WebRTC. Модуль трансляций включает страницу списка трансляций, страницу ведущего с управлением эфиром и страницу зрителя с видеоплеером и текстовым чатом.

На шестом этапе были реализованы панели управления для специалистов и администратора, каталог видеокурсов с механизмом покупки и просмотра, личный кабинет пользователя и страница истории покупок.

На седьмом этапе была выполнена адаптивная вёрстка всех страниц приложения с использованием адаптивных модификаторов Tailwind CSS, реализовано мобильное бургер-меню с анимацией и добавлен компонент согласия на обработку персональных данных.

4.2.2 Определение операций пользователей и составление функциональных блоков

Определение операций пользователей и их группировка в функциональные блоки позволяют формализовать взаимодействие каждой роли с системой и установить чёткие границы ответственности между модулями приложения. Данная систематизация необходима для обеспечения полноты реализованной функциональности и её соответствия сформулированным требованиям.

Операции незарегистрированного пользователя включают просмотр главной страницы и страниц описания услуг, просмотр каталога товаров интернет-магазина, просмотр списка предстоящих трансляций, просмотр каталога видеокурсов, просмотр руководства пользователя, переход к регистрации и авторизации. Данные операции не требуют отправки аутентифицированных запросов к серверной части.

Операции зарегистрированного пользователя включают все операции незарегистрированного пользователя, а также авторизацию и выход из системы, просмотр и редактирование профиля, запись на приём к психологу с выбором слота, запись на диагностику с выбором слота, запись на сеанс тибетских чаш с выбором города и слота, вход в видеокомнату для дистанционной консультации, добавление товаров в корзину и оформление заказа, покупку и просмотр видеокурсов, подключение к прямой трансляции и участие в чате, просмотр истории покупок и записей.

Операции специалиста (психолога, диагноста) включают все операции зарегистрированного пользователя, а также управление собственным расписанием (добавление и удаление слотов), просмотр списка записей с контактными данными клиентов, создание и управление видеокомнатами для консультаций, просмотр журнала активности и списка пользователей.

Операции специалиста по тибетским чашам включают управление расписанием с указанием города, просмотр записей с контактными данными клиентов (ФИО, телефон, email), просмотр списка пользователей и журнала активности. Видеокомнаты для данной роли недоступны.

Операции администратора включают все операции специалиста, а также управление пользователями и назначение ролей, управление каталогом товаров (создание, редактирование, удаление), управление видеокурсами (создание, загрузка видеоуроков, удаление), управление трансляциями (создание, запуск эфира, остановка), управление лабораториями, управление руководством пользователя, просмотр сводной статистики платформы.

4.2.3 Проектирование структуры экранов и схемы навигации

Проектирование структуры экранов и схемы навигации определяет, каким образом пользователи перемещаются между различными разделами приложения, сколько экранных форм предусмотрено в системе и какие переходы между ними доступны. Навигационная схема является связующим звеном между функциональными блоками и конкретной реализацией интерфейса, обеспечивая логичную и предсказуемую последовательность действий для каждой роли пользователя.

Клиентская часть веб-ресурса «Коосмо» включает более двадцати экранных форм, объединённых единой навигационной структурой. Точкой входа для всех пользователей служит главная страница, содержащая навигационную панель с ссылками на основные разделы. Навигационная панель является общим элементом, присутствующим на каждой странице приложения, и обеспечивает переход к любому разделу из любого текущего расположения.

Структура экранов выстроена по принципу функциональных групп. Группа аутентификации включает экран авторизации и экран регистрации, связанные взаимными переходами. Группа услуг объединяет страницы описания каждого направления (психология, диагностика, тибетские чаши, анализы) и соответствующие страницы записи на приём. Группа магазина включает каталог товаров и страницу корзины с последовательным переходом от просмотра к оформлению заказа. Группа образования включает каталог курсов и страницу просмотра отдельного курса с видеоуроками. Группа трансляций включает список эфиров и страницу просмотра трансляции с чатом. Личный кабинет является центральным агрегирующим экраном, предоставляющим доступ к персональным данным, записям и приобретённым ресурсам.

Панели управления специалистов и администратора реализованы как отдельные экранные формы с внутренней вкладочной навигацией. Переход к видеокомнате осуществляется из панели специалиста или из личного кабинета клиента по нажатию соответствующей кнопки.

4.2.4 Разработка дизайна интерфейса

Разработка дизайна интерфейса определяет визуальный стиль приложения, цветовую схему, типографику и принципы компоновки элементов на экранных формах. Дизайн интерфейса оказывает непосредственное влияние на первое впечатление пользователя и формирует его доверие к сервису.

Визуальный стиль веб-ресурса «Коосмо» выдержан в тёплой природной палитре, соответствующей тематике оздоровительного центра. Основными цветами являются мягкий бежевый (#F5F0EB) для фона, тёплый коричневый (#8B7355) для акцентных элементов и заголовков, белый (#FFFFFF) для карточек и контейнеров, а тёмно-серый (#333333) для основного текста. Данная цветовая схема создаёт ощущение спокойствия, натуральности и доверия, что соответствует позиционированию оздоровительного центра.

Типографика построена на использовании системных шрифтов без засечек, обеспечивающих высокую читаемость на экранах различных размеров и не требующих загрузки дополнительных файлов. Размер основного текста составляет 16 пикселей, заголовки первого уровня — 32 пикселя, второго уровня — 24 пикселя. Межстрочный интервал установлен на уровне 1.5, что обеспечивает комфортное чтение длинных текстовых блоков.

Элементы управления (кнопки, поля ввода, переключатели) имеют скруглённые углы радиусом 8–12 пикселей, что визуально смягчает интерфейс. Кнопки основных действий окрашены в акцентный коричневый цвет с белым текстом и имеют эффект затемнения при наведении курсора. Поля ввода выделены тонкой рамкой и имеют внутреннюю тень при фокусе, что обеспечивает визуальную обратную связь при взаимодействии.

Карточки товаров, услуг и курсов реализованы как прямоугольные контейнеры с изображением в верхней части, текстовым описанием в центре и кнопкой действия в нижней части. Карточки имеют лёгкую тень, визуально отделяющую их от фона, и увеличивают тень при наведении, создавая эффект приподнятости.

На мобильных устройствах (ширина экрана менее 768 пикселей) навигационная панель заменяется кнопкой бургер-меню в правом верхнем углу экрана. При нажатии открывается полноэкранное выдвижное меню с анимацией появления, содержащее все навигационные ссылки, расположенные вертикальным списком. Сетки карточек перестраиваются в одноколоночный формат, таблицы получают горизонтальную прокрутку, а размеры интерактивных элементов увеличиваются для комфортного использования на сенсорных экранах.

4.3 Входные, выходные и промежуточные данные

Описание входных, выходных и промежуточных данных определяет, какую информацию клиентская часть приложения получает извне, какую информацию формирует в процессе работы и какую информацию передаёт серверной части или отображает пользователю. Корректное определение потоков данных является необходимым условием для обеспечения целостности и согласованности работы системы.

Входные данные клиентской части можно разделить на две категории: данные, вводимые пользователем через формы интерфейса, и данные, получаемые от серверной части через API.

Данные, вводимые пользователем, включают учётные данные при регистрации (логин, пароль, ФИО, телефон, электронная почта), учётные данные при авторизации (логин, пароль), контактные данные при записи на приём (ФИО, телефон), параметры товаров при редактировании администратором (наименование, описание, цена, категория, изображение), параметры трансляций (заголовок, описание, дата, время, спикер, стоимость, превью), параметры курсов (заголовок, описание, цена, изображение, видеоуроки), текстовые сообщения в чате трансляций.

Данные, получаемые от серверной части, включают информацию о текущем пользователе (идентификатор, логин, роль, ФИО, телефон, email), списки товаров, курсов, трансляций, слотов расписания, записей, пользователей, статистику и журнал активности.

Промежуточные данные формируются клиентской частью в процессе работы и хранятся в оперативной памяти браузера посредством состояний React-компонентов. К ним относятся текущее состояние авторизации пользователя, содержимое корзины до оформления заказа, значения полей форм до их отправки, состояние фильтров и поисковых запросов, параметры WebRTC-соединения (объекты RTCPeerConnection, MediaStream), флаги состояния камеры и микрофона.

Выходные данные клиентской части представляют собой визуальное отображение информации на экране пользователя (страницы, карточки, таблицы, формы, видеопотоки), HTTP-запросы к серверной части с данными пользовательских действий (запросы на создание, чтение, обновление и удаление ресурсов), а также сигнальные сообщения WebRTC, передаваемые через сервер для установления видеосвязи.

4.4 Реализация используемых методов хранения, обработки и передачи информации об объектах предметной области

Данный подраздел описывает конкретные алгоритмы и технические решения, применённые при реализации методов хранения, обработки и передачи данных, описанных ранее в пункте 2.1.3. Детальное описание алгоритмов позволяет воспроизвести логику работы системы и обеспечить её корректное сопровождение в дальнейшем.

4.4.1 Методы хранения данных на стороне клиента

В пункте 2.1.3.1 были определены основные способы хранения информации на стороне клиента: состояние React-компонентов, локальное хранилище браузера и cookie-файлы. В данном подразделе описываются алгоритмы практической реализации каждого из перечисленных способов.

Управление оперативными данными в компонентах реализовано посредством хука useState. При монтировании страничного компонента (срабатывании хука useEffect с пустым массивом зависимостей) выполняется асинхронный запрос к серверу для получения актуальных данных. Полученные данные записываются в состояние компонента, что вызывает его перерисовку. При выполнении пользовательских действий (нажатие кнопки, отправка формы) состояние обновляется как оптимистично (немедленное изменение интерфейса), так и с подтверждением от сервера (повторная загрузка данных после успешного ответа).

Алгоритм работы с localStorage для хранения согласия на обработку данных реализован в компоненте CookieConsent. При монтировании компонент проверяет наличие ключа «cookieConsent» в localStorage. Если значение отсутствует, отображается баннер с текстом уведомления и кнопкой «Принимаю». При нажатии кнопки в localStorage записывается значение «true», и баннер скрывается. При повторных визитах компонент обнаруживает сохранённое значение и не отображает баннер.

Cookie с токеном аутентификации устанавливается серверной частью при успешном входе пользователя с флагом httpOnly, что исключает доступ к токену из клиентского JavaScript-кода. Клиентская часть не управляет содержимым данного cookie напрямую, однако обеспечивает его автоматическую отправку с каждым запросом посредством параметра credentials: "include" в конфигурации Fetch API.

4.4.2 Алгоритмы реализации используемых математических моделей

В пункте 2.1.3.2 были описаны математические модели, применяемые для обработки информации на стороне клиента: валидация ввода, вычисление стоимости корзины, форматирование дат и определение доступности видеокомнаты. В данном подразделе приводятся алгоритмы их программной реализации.

Алгоритм валидации формы регистрации выполняется в обработчике события отправки формы (handleSubmit). На первом шаге проверяется заполненность обязательных полей — логина, пароля, ФИО и телефона. Если хотя бы одно из полей пустое, в состояние компонента записывается сообщение об ошибке и выполнение прерывается. На втором шаге проверяется совпадение пароля с его подтверждением. При несовпадении отображается соответствующее сообщение. Только после прохождения всех проверок данные отправляются на сервер.

Алгоритм вычисления итоговой суммы корзины реализован как редуктор массива товаров: total = items.reduce((sum, item) => sum + item.price * item.quantity, 0). Вычисление выполняется при каждом рендере компонента корзины и используется для отображения итоговой суммы. Для форматирования результата в денежном формате применяется приведение числа к строке с двумя десятичными знаками.

Алгоритм форматирования даты извлекает год, месяц, число, часы и минуты из строки даты, полученной от сервера, и формирует строку в формате «ДД.ММ.ГГГГ ЧЧ:ММ» с использованием вспомогательной функции formatDate. Для группировки слотов расписания по датам слоты сортируются по дате и объединяются в именованные группы, где ключом является строковое представление даты без времени.

Алгоритм определения доступности кнопки входа в видеокомнату вычисляет разницу между временем начала записи и текущим временем: diff = appointmentTime - Date.now(). Если diff меньше 7 200 000 миллисекунд (два часа) и больше нуля, кнопка «Войти в комнату» становится активной. После истечения времени записи кнопка скрывается.

4.4.3 Алгоритмы применения методов визуального представления данных

В пункте 2.1.3.3 были описаны способы визуального представления информации: карточный интерфейс, табличный формат, списковый формат. В данном подразделе описываются алгоритмы их программной реализации в клиентской части.

Алгоритм отображения каталога товаров с фильтрацией реализован следующим образом. При загрузке страницы компонент получает полный список товаров от сервера и сохраняет его в состоянии. Пользователь выбирает категорию нажатием на соответствующую кнопку-фильтр. Активная категория записывается в состояние компонента. При рендере массив товаров фильтруется: если выбрана категория «Все товары», отображается полный список; в противном случае отображаются только товары, поле категории которых совпадает с выбранным значением. Результат фильтрации отображается в виде адаптивной сетки карточек, перестраивающейся с помощью Tailwind CSS от одной колонки на мобильных до трёх-четырёх на десктопных экранах.

Алгоритм отображения расписания с группировкой по датам получает от сервера массив слотов и выполняет их группировку: для каждого слота извлекается дата (без времени), и слот добавляется в соответствующую группу. Группы отображаются последовательно: заголовок с датой, затем список временных слотов с индикацией статуса (свободен или забронирован) и кнопкой записи для свободных слотов.

Алгоритм отображения административных таблиц строит HTML-таблицу на основе массива данных, полученных от сервера. Каждая строка таблицы соответствует одному объекту (пользователю, записи, товару). Таблица обёрнута в контейнер с горизонтальной прокруткой для корректного отображения на узких экранах. Для таблиц с большим количеством записей реализован поиск: при вводе текста в поле поиска массив данных фильтруется по совпадению с полями ФИО, логина, телефона или электронной почты.

4.4.4 Алгоритмы использования технологий передачи данных

В пункте 2.1.3.4 были описаны технологии получения и передачи информации: REST API через HTTP, WebRTC для видеосвязи, multipart/form-data для файлов. В данном подразделе приводятся алгоритмы их реализации в клиентской части.

В клиентской части разработан централизованный сервисный модуль API, представляющий собой объект с методами, сгруппированными по предметным областям: auth (авторизация), user (пользователь), products (товары), cart (корзина), streams (трансляции), courses (курсы), schedule (расписание), appointments (записи), room (видеокомнаты), bowlsSchedule (расписание тибетских чаш), diagnostics (диагностика), admin (администрирование), activity (журнал активности). Каждый метод инкапсулирует выполнение HTTP-запроса с указанием URL, метода, заголовков и тела запроса.

Алгоритм выполнения типичного запроса к серверной части состоит из следующих шагов:

1 Компонент вызывает соответствующий метод сервисного модуля, передавая необходимые параметры.

2 Метод формирует запрос с помощью Fetch API, указывая URL серверного маршрута, HTTP-метод (GET, POST, PUT, DELETE), заголовок Content-Type для JSON-данных и параметр credentials: "include" для автоматической отправки cookie с токеном.

3 Полученный ответ сервера передаётся в универсальную функцию обработки ошибок errorHandler, которая проверяет код статуса ответа. Если статус отличен от 200, функция извлекает сообщение об ошибке из тела ответа и выбрасывает исключение.

4 При успешном ответе данные извлекаются из тела ответа в формате JSON и возвращаются вызывающему компоненту.

5 Компонент обновляет своё состояние полученными данными, что инициирует перерисовку интерфейса с отображением актуальной информации.

В случае загрузки файлов (изображения товаров, видеозаписи курсов, превью трансляций) вместо JSON используется формат multipart/form-data. Данные упаковываются в объект FormData, к которому последовательно добавляются бинарный файл и сопутствующие текстовые поля. Для загрузки больших файлов (видеозаписи трансляций) используется XMLHttpRequest вместо Fetch API, что позволяет отслеживать прогресс загрузки через событие upload.onprogress и отображать пользователю индикатор выполнения.

4.4.5 Алгоритмы применения технологий видеосвязи и трансляций

Реализация видеосвязи и прямых трансляций является технически наиболее сложным элементом клиентской части и требует применения специализированных алгоритмов для установления и поддержания соединения, управления медиапотоками и обработки сетевых событий.

Алгоритм установления видеосвязи в комнате консультации состоит из следующих шагов:

1 При входе пользователя в видеокомнату клиентское приложение запрашивает у браузера доступ к камере и микрофону посредством вызова navigator.mediaDevices.getUserMedia, получая объект MediaStream с локальным видео- и аудиопотоком.

2 Создаётся объект RTCPeerConnection с конфигурацией STUN-сервера (stun:stun.l.google.com:19302), обеспечивающего определение внешнего сетевого адреса пользователя.

3 Треки локального медиапотока (видео и аудио) добавляются в объект соединения посредством метода addTrack.

4 Инициирующая сторона (специалист) создаёт предложение (offer) посредством вызова createOffer, устанавливает его как локальное описание сессии (setLocalDescription) и отправляет через серверную часть удалённой стороне.

5 Удалённая сторона (клиент) получает предложение, устанавливает его как удалённое описание (setRemoteDescription), создаёт ответ (answer) посредством createAnswer, устанавливает его как локальное описание и отправляет обратно.

6 В процессе согласования обе стороны обмениваются ICE-кандидатами через сервер для определения оптимального маршрута передачи данных.

7 После успешного согласования параметров устанавливается прямое одноранговое соединение, и медиапотоки начинают передаваться между участниками.

8 Пользователь имеет возможность отключать и включать микрофон и камеру посредством управления свойством enabled соответствующих треков, а также активировать демонстрацию экрана через getDisplayMedia с заменой видеотрека в соединении.

Алгоритм прямой трансляции основан на аналогичных принципах, однако имеет ряд отличий. Ведущий (администратор) запускает эфир и становится единственным источником медиапотока. При подключении каждого нового зрителя создаётся отдельное RTCPeerConnection-соединение между ведущим и этим зрителем. Параллельно функционирует текстовый чат: клиентская часть периодически (раз в две секунды) запрашивает новые сообщения у серверной части и отображает их в интерфейсе, обеспечивая общение зрителей и ведущего во время эфира.

4.5 Описание архитектурного решения

4.5.1 Структурная организация программной системы

Описание структурной организации программной системы раскрывает высокоуровневую архитектуру приложения, определяя взаимосвязи между основными подсистемами и принципы их взаимодействия. Понимание общей структуры системы необходимо как для текущей разработки, так и для дальнейшего масштабирования и сопровождения продукта.

Разрабатываемый веб-ресурс «Коосмо» построен на основе клиент-серверной архитектуры. Клиентская часть представляет собой одностраничное приложение, выполняемое в браузере пользователя и взаимодействующее с серверной частью посредством REST API по протоколу HTTP. Серверная часть предоставляет набор программных интерфейсов для управления данными и хранит информацию в реляционной базе данных. Клиентская и серверная части разработаны как независимые компоненты, связанные исключительно через программный интерфейс, что обеспечивает возможность их независимого развёртывания, тестирования и масштабирования.

В процессе разработки клиентская часть функционирует на порту 3000, а серверная — на порту 3001. Для решения проблемы межсайтовых запросов в среде разработки используется механизм проксирования, встроенный в Create React App: все запросы, не адресованные статическим файлам клиентской части, автоматически перенаправляются на серверную часть. В производственном окружении клиентская часть обслуживается тем же сервером, что и API, либо располагается за обратным прокси-сервером.

4.5.2 Архитектура программного кода

Описание архитектуры программного кода определяет принципы организации файловой структуры, правила именования и ответственности модулей, а также подходы к управлению состоянием и потоками данных внутри клиентской части приложения. Продуманная архитектура кода является залогом его читаемости, тестируемости и удобства сопровождения.

Клиентская часть организована в соответствии с компонентным подходом, принятым в экосистеме React. Файловая структура проекта разделена на следующие каталоги:

1 src/components — переиспользуемые компоненты интерфейса, не привязанные к конкретному маршруту: Header (навигационная панель с адаптивным бургер-меню), LoadingScreen (экран загрузки при инициализации приложения), CookieConsent (уведомление о согласии на обработку данных).

2 src/views — страничные компоненты, каждый из которых соответствует определённому маршруту приложения: Home, LoginView, RegistrationView, ShopPage, CartPage, ProfilePage, BookingPage, VideoRoomPage, StreamsPage, CoursesPage, AdminPage, PsychologistPanel, BowlsSpecialistPanel и другие.

3 src/services — сервисные модули для взаимодействия с серверной частью. Основной модуль api.ts экспортирует объект API с методами, сгруппированными по предметным областям, и определяет типы данных для запросов.

4 src/views/HomePage — компоненты и ресурсы, относящиеся к главной странице: основной компонент Frame с разметкой лендинга, изображения и иконки.

Каждый страничный компонент реализован как функциональный компонент React и использует хук useState для управления локальным состоянием (данные форм, списки, флаги загрузки, сообщения об ошибках), хук useEffect для выполнения побочных эффектов при монтировании компонента (загрузка данных с сервера, проверка авторизации), хук useNavigate из React Router для программной навигации и хук useParams для извлечения динамических параметров из URL.

Корневой компонент App определяет дерево маршрутов приложения с использованием компонентов Routes и Route из React Router DOM. Все маршруты обёрнуты в компонент PageTransition, обеспечивающий плавную анимацию появления страницы при навигации. Компонент Layout служит общей обёрткой, через которую вложенные маршруты отображаются посредством компонента Outlet.

5 ТЕСТИРОВАНИЕ И ОПТИМИЗАЦИЯ

5.1 План тестирования

Тестирование является неотъемлемой частью процесса разработки программного обеспечения, обеспечивающей выявление и устранение дефектов до передачи продукта в эксплуатацию. Грамотно составленный план тестирования определяет объём проверок, методы их проведения и критерии приёмки, что позволяет систематизировать процесс контроля качества и гарантировать соответствие реализованного функционала сформулированным требованиям.

Для клиентской части веб-ресурса «Коосмо» был разработан план тестирования, включающий несколько уровней проверки.

Функциональное тестирование направлено на проверку корректности выполнения каждой из заявленных функций приложения. В рамках данного уровня проверяется работоспособность регистрации и авторизации с различными комбинациями входных данных, корректность отображения каталога товаров и фильтрации по категориям, работоспособность механизма корзины и оформления заказов, корректность записи на приём к каждому типу специалиста с проверкой автозаполнения форм, работоспособность видеокомнат и прямых трансляций, корректность работы панелей управления специалистов и администратора, а также корректность отображения и редактирования профиля пользователя.

Тестирование пользовательского интерфейса направлено на проверку визуального соответствия реализованного интерфейса дизайн-макету, корректности отображения элементов на различных разрешениях экрана и в различных браузерах. В рамках данного уровня проверяется работоспособность адаптивной вёрстки на мобильных устройствах, корректность трансформации навигационного меню в бургер-меню, читаемость текстов и достаточность размеров интерактивных элементов на сенсорных экранах.

Кроссбраузерное тестирование предполагает проверку работоспособности приложения в актуальных версиях браузеров Google Chrome, Mozilla Firefox, Microsoft Edge и Safari. Особое внимание уделяется корректности работы WebRTC-модулей, поскольку реализация данной технологии может отличаться в различных браузерах.

Тестирование безопасности включает проверку невозможности доступа к защищённым разделам без авторизации, корректности работы механизма аутентификации на основе cookie, отображения уведомления о согласии на обработку персональных данных и невозможности извлечения токена аутентификации из JavaScript-кода.

Нагрузочное тестирование направлено на оценку поведения клиентской части при одновременном подключении нескольких зрителей к прямой трансляции и при работе с большим количеством товаров в каталоге интернет-магазина.

5.2 Результаты тестирования и оптимизация

Представление результатов тестирования позволяет зафиксировать выявленные в ходе проверок дефекты, описать предпринятые меры по их устранению и подтвердить итоговое соответствие продукта установленным требованиям. Результаты тестирования служат доказательной базой готовности программного средства к эксплуатации.

По итогам проведённого тестирования были получены следующие результаты.

Функциональное тестирование подтвердило корректность работы всех заявленных функций. Регистрация пользователя с заполнением обязательных полей (ФИО, телефон, логин, пароль) выполняется успешно, при этом попытка регистрации с существующим логином корректно возвращает сообщение об ошибке. Авторизация устанавливает сессию и обновляет состояние интерфейса, отображая элементы управления авторизованного пользователя. Каталог товаров корректно загружается и фильтруется по всем шести категориям. Механизм корзины обеспечивает добавление, изменение количества, удаление товаров и оформление заказа. Запись на приём к психологу, диагносту и специалисту по тибетским чашам выполняется корректно с автозаполнением контактных данных из профиля. Видеокомната обеспечивает двустороннюю видеосвязь с возможностью управления камерой, микрофоном и демонстрацией экрана. Прямые трансляции обеспечивают передачу видеопотока от ведущего зрителям с одновременной работой текстового чата.

В ходе тестирования были выявлены и устранены следующие дефекты. При использовании проксирования в среде разработки cookie-файлы с токеном аутентификации не передавались серверу из-за различия портов клиентской и серверной частей, что было решено настройкой параметра proxy в конфигурации Create React App и использованием пустого базового URL для API-запросов. При одновременном подключении к видеокомнате обоих участников до завершения сигнального обмена возникала ситуация гонки, устранённая введением механизма определения инициатора соединения на основе идентификатора роли специалиста.

Тестирование адаптивного интерфейса подтвердило корректное отображение всех страниц на устройствах с шириной экрана от 320 до 2560 пикселей. Бургер-меню корректно открывается и закрывается на мобильных устройствах, блокируя прокрутку основного содержимого. Таблицы в панелях специалистов и администратора поддерживают горизонтальную прокрутку на узких экранах.

Кроссбраузерное тестирование подтвердило работоспособность приложения в Google Chrome версии 120 и выше, Mozilla Firefox версии 121 и выше, Microsoft Edge версии 120 и выше и Safari версии 17 и выше. WebRTC-модули функционируют корректно во всех перечисленных браузерах.

6 РУКОВОДСТВО ПОЛЬЗОВАТЕЛЯ

Руководство пользователя представляет собой описание основных действий, которые пользователь может выполнять при работе с веб-ресурсом «Коосмо». Данный раздел адресован конечным пользователям платформы и описывает последовательность шагов для выполнения наиболее востребованных операций.

Для начала работы с веб-ресурсом необходимо открыть один из поддерживаемых браузеров (Google Chrome, Mozilla Firefox, Microsoft Edge или Safari) и перейти по адресу сайта. Главная страница содержит навигационное меню в верхней части экрана со ссылками на основные разделы: «Диагностика», «Анализы», «Самомассаж», «Трансляции», «Тибетские чаши», «Психология». При просмотре с мобильного устройства навигационное меню доступно через кнопку с тремя горизонтальными линиями в правом верхнем углу экрана.

Для регистрации необходимо нажать кнопку «Регистрация» в правом верхнем углу экрана и заполнить форму, указав ФИО, номер телефона, адрес электронной почты (необязательно), логин и пароль. После нажатия кнопки «Зарегистрироваться» система создаст учётную запись и предложит перейти к авторизации. Для входа в систему необходимо нажать кнопку «Вход», ввести логин и пароль и нажать «Войти». После успешного входа в навигационном меню появятся ссылки на личный кабинет и корзину.

Для записи на психологическую консультацию необходимо перейти в раздел «Психология» через навигационное меню. На странице отображается информация о стоимости консультации и доступные временные слоты, сгруппированные по датам. Для записи следует нажать кнопку «Записаться» напротив выбранного времени. В открывшемся окне будут автоматически заполнены ФИО и телефон из профиля, при необходимости их можно изменить. После нажатия кнопки «Подтвердить» запись будет оформлена, и информация о ней появится в личном кабинете. В назначенное время в личном кабинете появится кнопка для входа в видеокомнату, где будет организована видеосвязь со специалистом.

Для записи на сеанс тибетских чаш необходимо перейти в раздел «Тибетские чаши». На странице отображаются доступные слоты с возможностью выбора города (Новосибирск или Бийск). Процедура записи аналогична записи к психологу. После записи специалист свяжется с клиентом по указанному телефону для согласования деталей.

Для записи на диагностику необходимо перейти в раздел «Диагностика» и выбрать интересующий тип диагностики: ногтей, языка, глаз, кожи или тела и осанки. Далее выбрать свободный слот и подтвердить запись. Диагностика проводится дистанционно через видеокомнату.

Для покупки товаров необходимо перейти в раздел «Магазин» через навигационное меню или ссылку «Самомассаж». В каталоге можно выбрать категорию товаров с помощью кнопок-фильтров в верхней части страницы. Для добавления товара в корзину следует нажать кнопку «В корзину» на карточке товара. Для перехода к оформлению необходимо нажать на иконку корзины в правом верхнем углу экрана. На странице корзины можно изменить количество товаров, удалить отдельные позиции или очистить корзину целиком. Для завершения покупки следует нажать кнопку «Оформить заказ».

Для участия в прямой трансляции необходимо перейти в раздел «Трансляции». На странице отображается список предстоящих и текущих эфиров. Для подключения к текущей трансляции следует нажать кнопку «Присоединиться». После подключения на экране появится видеопоток ведущего и текстовый чат, в котором можно задавать вопросы и общаться с другими участниками.

Для покупки и просмотра видеокурсов необходимо перейти в раздел курсов, выбрать интересующий курс и нажать кнопку «Купить». После покупки курс станет доступен для просмотра. Для перехода к урокам следует открыть курс в личном кабинете или на странице каталога курсов.

Личный кабинет доступен по нажатию на иконку профиля в навигационном меню. В кабинете отображаются персональные данные с возможностью редактирования, список предстоящих записей к специалистам, история записей на диагностику и тибетские чаши, а также подключённые трансляции. Для изменения персональных данных следует заполнить соответствующие поля и нажать кнопку «Сохранить».

При первом посещении сайта в нижней части экрана появится уведомление о сборе и обработке персональных данных. Для продолжения работы необходимо нажать кнопку «Принимаю».


ЗАКЛЮЧЕНИЕ

В ходе выполнения данной выпускной квалификационной работы была разработана клиентская часть информационного веб-ресурса «Коосмо» — многофункциональной онлайн-платформы для оздоровительного центра, объединяющей в себе систему онлайн-записи к специалистам, встроенную видеосвязь для дистанционных консультаций, модуль прямых трансляций с текстовым чатом, интернет-магазин с корзиной и оформлением заказов, каталог видеокурсов с возможностью покупки и просмотра, а также панели управления для специалистов и администратора.

На этапе анализа требований были определены бизнес-цели проекта, направленные на увеличение клиентского трафика, рост продаж услуг и товаров, расширение географии предоставления услуг за счёт видеоконсультаций и укрепление бренда посредством прямых трансляций и образовательного контента. Были сформулированы пользовательские, системные и функциональные требования, определены классы пользователей и описаны типовые сценарии взаимодействия с системой. Проведённый обзор существующих программных реализаций подтвердил обоснованность разработки собственного решения, поскольку ни один из рассмотренных аналогов не предоставляет комплексной функциональности, включающей видеосвязь, трансляции, интернет-магазин и образовательный портал в рамках единой платформы.

На этапе проектирования был выполнен сравнительный анализ технологий фронтенд-разработки, по результатам которого в качестве основного инструментария были выбраны TypeScript, React, Tailwind CSS и React Router DOM. Данный технологический стек обеспечил высокую скорость разработки, строгую типизацию кода, адаптивность интерфейса и модульную архитектуру приложения.

На этапе реализации были разработаны все запланированные функциональные модули: система аутентификации с регистрацией и авторизацией, главная страница с представлением услуг, интернет-магазин с фильтрацией по категориям и управлением корзиной, три независимых модуля записи на приём с автозаполнением форм данными профиля, модуль видеокомнат на основе WebRTC для двусторонней видеосвязи, модуль прямых трансляций с текстовым чатом, каталог видеокурсов, личный кабинет пользователя, панели управления для психолога, диагноста, специалиста по тибетским чашам и администратора. Реализована полноценная адаптивная вёрстка с поддержкой мобильных устройств, включая бургер-меню и горизонтальную прокрутку таблиц.

На этапе тестирования было проведено 88 проверок, охватывающих функциональное, интерфейсное, кроссбраузерное тестирование, а также тестирование безопасности и нагрузки. Все выявленные дефекты были устранены, и по результатам повторного тестирования все проверки пройдены успешно.

Особенностью разработанного решения является встроенная система видеокомнат на основе технологии WebRTC, обеспечивающая проведение дистанционных консультаций без использования сторонних сервисов, что особенно актуально в условиях периодических ограничений доступа к популярным средствам видеосвязи на территории Российской Федерации.

Разработанная клиентская часть полностью соответствует сформулированным требованиям и готова к интеграции с серверной частью для последующего развёртывания в производственном окружении. Дальнейшее развитие проекта может включать реализацию системы онлайн-оплаты, интеграцию с внешними сервисами уведомлений, расширение функциональности панели аналитики и добавление мультиязычной поддержки.

// src/services/api.ts

export const BASE_URL = "";

type LoginData = {
  login: string;
  password: string;
};

type RegistrationData = {
  login: string;
  password: string;
  fullName?: string;
  phone?: string;
  email?: string;
};

const errorHandler = async (response: Response) => {
  if (response.status !== 200) {
    const responseData = await response.json();
    throw Error(responseData.message);
  }
};

export const API = {
  auth: {
    login: async (data: LoginData) => {
      const response = await fetch(`${BASE_URL}/auth`, {
        method: "POST",
        credentials: "include",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(data),
      });
      await errorHandler(response);
    },
    logout: async () => {
      const response = await fetch(`${BASE_URL}/auth`, {
        method: "DELETE",
        credentials: "include",
      });
      await errorHandler(response);
    },
  },
  user: {
    register: async (data: RegistrationData) => {
      const response = await fetch(`${BASE_URL}/user`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(data),
      });
      await errorHandler(response);
    },
    getCurrentUser: async () => {
      const response = await fetch(`${BASE_URL}/user`, {
        credentials: "include",
        method: "GET",
      });
      await errorHandler(response);
      return await response.json();
    },
    updateProfile: async (data: { fullName: string; phone: string; email?: string }) => {
      const response = await fetch(`${BASE_URL}/user/profile`, {
        method: "PUT",
        credentials: "include",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(data),
      });
      await errorHandler(response);
      return await response.json();
    },
  },
  // ... остальные модули: products, cart, streams, courses,
  // schedule, appointments, room, bowlsSchedule, diagnostics,
  // admin, activity, bowlsMedia, guide, labs, settings, streamRoom
};

Компонент страницы регистрации (src/views/RegistrationView.tsx)
Компонент отвечает за регистрацию нового пользователя с валидацией полей на стороне клиента.

import React, { useState, FormEvent } from "react";
import { Link, useNavigate } from "react-router-dom";
import { API } from "../services/api";

const RegistrationView: React.FC = () => {
  const navigate = useNavigate();
  const [login, setLogin] = useState("");
  const [password, setPassword] = useState("");
  const [confirmPassword, setConfirmPassword] = useState("");
  const [fullName, setFullName] = useState("");
  const [phone, setPhone] = useState("");
  const [email, setEmail] = useState("");
  const [error, setError] = useState("");
  const [success, setSuccess] = useState("");
  const [submitting, setSubmitting] = useState(false);

  const handleSubmit = async (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    setError("");
    setSuccess("");
    if (!login.trim() || !password.trim()) {
      setError("Заполните логин и пароль");
      return;
    }
    if (!fullName.trim()) {
      setError("Укажите ФИО");
      return;
    }
    if (!phone.trim()) {
      setError("Укажите номер телефона");
      return;
    }
    if (password !== confirmPassword) {
      setError("Пароли не совпадают");
      return;
    }
    setSubmitting(true);
    try {
      await API.user.register({
        login: login.trim(),
        password,
        fullName: fullName.trim(),
        phone: phone.trim(),
        email: email.trim() || undefined,
      });
      setSuccess("Регистрация выполнена!");
    } catch (e) {
      if (e instanceof Error) setError(e.message);
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <div className="min-h-screen flex items-center justify-center bg-[#F5F0EB]">
      <form onSubmit={handleSubmit}
        className="bg-white rounded-2xl shadow-lg p-8 w-full max-w-md">
        <h2 className="text-2xl font-bold text-center mb-6">Регистрация</h2>
        {error && <div className="text-red-500 mb-4">{error}</div>}
        {success && <div className="text-green-600 mb-4">{success}
          <Link to="/login" className="underline ml-2">Войти</Link>
        </div>}
        <input type="text" placeholder="ФИО" value={fullName}
          onChange={(e) => setFullName(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <input type="tel" placeholder="Телефон" value={phone}
          onChange={(e) => setPhone(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <input type="email" placeholder="Email (необязательно)" value={email}
          onChange={(e) => setEmail(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <input type="text" placeholder="Логин" value={login}
          onChange={(e) => setLogin(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <input type="password" placeholder="Пароль" value={password}
          onChange={(e) => setPassword(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <input type="password" placeholder="Подтвердите пароль"
          value={confirmPassword}
          onChange={(e) => setConfirmPassword(e.target.value)}
          className="w-full border rounded-lg p-3 mb-3" />
        <button type="submit" disabled={submitting}
          className="w-full bg-[#8B7355] text-white rounded-lg p-3
          hover:bg-[#7a644a] transition">
          {submitting ? "Отправка..." : "Зарегистрироваться"}
        </button>
        <p className="text-center mt-4 text-sm">
          Уже есть аккаунт? <Link to="/login" className="underline">Войти</Link>
        </p>
      </form>
    </div>
  );
};

export default RegistrationView;

Компонент страницы видеокомнаты (src/views/VideoRoomPage.tsx, фрагмент)
Фрагмент компонента, реализующего двустороннюю видеосвязь на основе WebRTC.

import React, { useEffect, useRef, useState } from "react";
import { useParams, useNavigate } from "react-router-dom";
import { API } from "../services/api";

const VideoRoomPage: React.FC = () => {
  const { roomId } = useParams<{ roomId: string }>();
  const navigate = useNavigate();
  const localVideoRef = useRef<HTMLVideoElement>(null);
  const remoteVideoRef = useRef<HTMLVideoElement>(null);
  const pcRef = useRef<RTCPeerConnection | null>(null);
  const localStreamRef = useRef<MediaStream | null>(null);
  const [micOn, setMicOn] = useState(true);
  const [camOn, setCamOn] = useState(true);
  const [connected, setConnected] = useState(false);
  const [user, setUser] = useState<any>(null);
  const [roomData, setRoomData] = useState<any>(null);

  useEffect(() => {
    const init = async () => {
      try {
        const currentUser = await API.user.getCurrentUser();
        setUser(currentUser);
        const room = await API.room.get(roomId!);
        setRoomData(room);
        const stream = await navigator.mediaDevices.getUserMedia({
          video: true, audio: true,
        });
        localStreamRef.current = stream;
        if (localVideoRef.current) {
          localVideoRef.current.srcObject = stream;
        }
        const pc = new RTCPeerConnection({
          iceServers: [{ urls: "stun:stun.l.google.com:19302" }],
        });
        pcRef.current = pc;
        stream.getTracks().forEach((track) => pc.addTrack(track, stream));
        pc.ontrack = (event) => {
          if (remoteVideoRef.current) {
            remoteVideoRef.current.srcObject = event.streams[0];
            setConnected(true);
          }
        };
        pc.onicecandidate = (event) => {
          if (event.candidate) {
            API.room.sendSignal(roomId!, {
              type: "candidate",
              candidate: event.candidate,
            });
          }
        };
        // ... логика сигнального обмена (offer/answer)
      } catch (e) {
        console.error("Ошибка инициализации видеокомнаты:", e);
      }
    };
    init();
    return () => {
      localStreamRef.current?.getTracks().forEach((t) => t.stop());
      pcRef.current?.close();
    };
  }, [roomId]);

  const toggleMic = () => {
    const audioTrack = localStreamRef.current?.getAudioTracks()[0];
    if (audioTrack) {
      audioTrack.enabled = !audioTrack.enabled;
      setMicOn(audioTrack.enabled);
    }
  };

  const toggleCam = () => {
    const videoTrack = localStreamRef.current?.getVideoTracks()[0];
    if (videoTrack) {
      videoTrack.enabled = !videoTrack.enabled;
      setCamOn(videoTrack.enabled);
    }
  };

  const shareScreen = async () => {
    try {
      const screenStream = await navigator.mediaDevices.getDisplayMedia({
        video: true,
      });
      const screenTrack = screenStream.getVideoTracks()[0];
      const sender = pcRef.current
        ?.getSenders()
        .find((s) => s.track?.kind === "video");
      if (sender) sender.replaceTrack(screenTrack);
      screenTrack.onended = () => {
        const camTrack = localStreamRef.current?.getVideoTracks()[0];
        if (camTrack && sender) sender.replaceTrack(camTrack);
      };
    } catch (e) {}
  };

  return (
    <div className="min-h-screen bg-gray-900 flex flex-col
      items-center justify-center p-4">
      <div className="flex flex-col md:flex-row gap-4 mb-6">
        <video ref={remoteVideoRef} autoPlay playsInline
          className="w-full md:w-[640px] rounded-lg bg-black" />
        <video ref={localVideoRef} autoPlay playsInline muted
          className="w-32 h-24 md:w-48 md:h-36 rounded-lg bg-black" />
      </div>
      <div className="flex space-x-4">
        <button onClick={toggleMic}
          className={`px-4 py-2 rounded-lg ${micOn
            ? "bg-gray-600" : "bg-red-600"} text-white`}>
          {micOn ? "Микрофон вкл" : "Микрофон выкл"}
        </button>
        <button onClick={toggleCam}
          className={`px-4 py-2 rounded-lg ${camOn
            ? "bg-gray-600" : "bg-red-600"} text-white`}>
          {camOn ? "Камера вкл" : "Камера выкл"}
        </button>
        <button onClick={shareScreen}
          className="px-4 py-2 rounded-lg bg-blue-600 text-white">
          Демонстрация экрана
        </button>
        <button onClick={() => navigate(-1)}
          className="px-4 py-2 rounded-lg bg-red-700 text-white">
          Завершить
        </button>
      </div>
    </div>
  );
};

export default VideoRoomPage;


Компонент страницы корзины (src/views/CartPage.tsx, фрагмент)
Фрагмент компонента, реализующего управление корзиной и вычисление итоговой суммы.

import React, { useEffect, useState } from "react";
import { useNavigate } from "react-router-dom";
import { API } from "../services/api";

interface CartItem {
  id: number;
  name: string;
  price: number;
  quantity: number;
  image?: string;
}

const CartPage: React.FC = () => {
  const navigate = useNavigate();
  const [items, setItems] = useState<CartItem[]>([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    API.cart.get()
      .then((data) => setItems(data))
      .catch(() => {})
      .finally(() => setLoading(false));
  }, []);

  const total = items.reduce(
    (sum, item) => sum + item.price * item.quantity, 0
  );

  const updateQuantity = async (id: number, quantity: number) => {
    if (quantity < 1) return;
    await API.cart.update(id, quantity);
    setItems((prev) =>
      prev.map((item) => (item.id === id ? { ...item, quantity } : item))
    );
  };

  const removeItem = async (id: number) => {
    await API.cart.remove(id);
    setItems((prev) => prev.filter((item) => item.id !== id));
  };

  const checkout = async () => {
    try {
      await API.cart.checkout();
      setItems([]);
      alert("Заказ оформлен!");
    } catch (e) {}
  };

  if (loading) return <div className="text-center mt-10">Загрузка...</div>;

  return (
    <div className="max-w-3xl mx-auto p-6">
      <h1 className="text-2xl font-bold mb-6">Корзина</h1>
      {items.length === 0 ? (
        <p>Корзина пуста</p>
      ) : (
        <>
          {items.map((item) => (
            <div key={item.id}
              className="flex items-center justify-between border-b py-4">
              <div>
                <p className="font-semibold">{item.name}</p>
                <p className="text-gray-500">{item.price} ₽</p>
              </div>
              <div className="flex items-center space-x-2">
                <button onClick={() => updateQuantity(item.id,
                  item.quantity - 1)}>−</button>
                <span>{item.quantity}</span>
                <button onClick={() => updateQuantity(item.id,
                  item.quantity + 1)}>+</button>
                <button onClick={() => removeItem(item.id)}
                  className="text-red-500 ml-4">Удалить</button>
              </div>
            </div>
          ))}
          <div className="mt-6 flex justify-between items-center">
            <p className="text-xl font-bold">Итого: {total} ₽</p>
            <button onClick={checkout}
              className="bg-[#8B7355] text-white px-6 py-3 rounded-lg">
              Оформить заказ
            </button>
          </div>
        </>
      )}
    </div>
  );
};

export default CartPage;
