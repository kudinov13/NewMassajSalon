ПЕРЕЧЕНЬ СОКРАЩЕНИЙ, УСЛОВНЫХ ОБОЗНАЧЕНИЙ И ТЕРМИНОВ

SPA (Single Page Application) — одностраничное приложение, в котором навигация и обновление контента происходят без полной перезагрузки страницы [1].

React — JavaScript-библиотека для построения пользовательских интерфейсов, разработанная компанией Meta (Facebook), основанная на компонентном подходе [2].

TypeScript — надмножество JavaScript с системой статической типизации, компилируемое в JavaScript [3].

JSX/TSX — расширение синтаксиса JavaScript/TypeScript, позволяющее описывать элементы интерфейса в декларативной HTML-подобной форме внутри кода.

React Router — библиотека клиентской маршрутизации для React, обеспечивающая навигацию между страницами SPA без перезагрузки [4].

React Hook — функция, позволяющая использовать состояние и другие возможности React в функциональных компонентах (useState, useEffect, useRef, useCallback и др.) [2].

REST API (Representational State Transfer) — архитектурный стиль взаимодействия клиента с сервером через HTTP-запросы [5].

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня для передачи данных в сети Интернет.

WebRTC (Web Real-Time Communication) — технология передачи потоковых аудио- и видеоданных между браузерами в режиме реального времени без промежуточных серверов [6].

RTCPeerConnection — браузерный API для установления и управления WebRTC-соединением между двумя узлами.

SDP (Session Description Protocol) — протокол описания параметров мультимедийной сессии в WebRTC.

ICE (Interactive Connectivity Establishment) — фреймворк для установления сетевого соединения в WebRTC.

Tailwind CSS — утилитарный CSS-фреймворк, предоставляющий набор предопределённых классов для стилизации элементов [7].

React Final Form — библиотека для управления формами в React с поддержкой валидации [8].

Yup — библиотека для описания схем валидации данных [9].

CORS (Cross-Origin Resource Sharing) — механизм разрешения кросс-доменных HTTP-запросов.

Компонент — независимый, повторно используемый блок пользовательского интерфейса в React, инкапсулирующий логику, разметку и стили.

Маршрут (Route) — правило сопоставления URL-адреса с конкретным компонентом в React Router.

Long Polling — техника периодического опроса сервера клиентом для получения новых данных.


ВВЕДЕНИЕ

В современном обществе наблюдается устойчивый рост интереса к альтернативным методам оздоровления, психологической поддержке и комплексному подходу к здоровью. Центры здоровья и красоты, предлагающие услуги психологического консультирования, тибетского массажа, диагностики и продажу натуральных продуктов, нуждаются в автоматизации бизнес-процессов и создании удобного цифрового интерфейса для взаимодействия с клиентами.

Актуальность данной работы обусловлена тем, что большинство подобных центров не имеют собственного веб-приложения, а используют разрозненные инструменты: социальные сети для продвижения, мессенджеры для записи, сторонние сервисы для видеоконсультаций. Это создаёт неудобства для клиентов и снижает профессиональный имидж организации.

Существующие решения (Yclients, DIKIDI, Bookly) предоставляют шаблонные интерфейсы, не адаптируемые под специфику многопрофильного центра здоровья. Они не поддерживают встроенную видеосвязь для дистанционных приёмов, не позволяют продавать видеокурсы, проводить прямые трансляции мастер-классов и не обеспечивают выбор города при записи на тибетский массаж. Данные ограничения обуславливают необходимость создания собственного клиентского приложения с уникальным пользовательским интерфейсом.

Целью данной выпускной квалификационной работы является разработка клиентской части (фронтенда) веб-приложения «Harmony Spa» — одностраничного приложения (SPA) на основе React и TypeScript, обеспечивающего полнофункциональный пользовательский интерфейс для всех категорий пользователей системы: клиентов, психолога, специалиста по тибетским чашам и администратора.

Для достижения поставленной цели определены следующие задачи:
— провести анализ требований к пользовательскому интерфейсу;
— спроектировать структуру SPA-приложения с клиентской маршрутизацией;
— реализовать компонентную архитектуру на React с TypeScript;
— реализовать взаимодействие с серверным REST API через модуль api.ts;
— реализовать видеоконференцсвязь на стороне клиента через WebRTC API;
— реализовать демонстрацию экрана в видеоконференциях;
— реализовать административные панели для каждой роли;
— стилизовать интерфейс с использованием Tailwind CSS;
— провести тестирование клиентской части.

Практическая значимость работы заключается в создании современного, адаптивного и интуитивно понятного пользовательского интерфейса, пригодного для использования реальным центром здоровья и красоты.


1 ПОСТАНОВКА ЗАДАЧИ ВКР

1.1 Бизнес-требования

Заказчиком разработки является центр здоровья и красоты «Harmony Spa», работающий в городах Новосибирск и Бийск. Центр предоставляет широкий спектр услуг: психологическое консультирование, массаж тибетскими чашами (уникальная процедура «в четыре руки»), визуальную диагностику, продажу натуральных продуктов и образовательных видеокурсов, проведение прямых трансляций мастер-классов.

Бизнес-цели, связанные с клиентской частью приложения:

1. Предоставление клиентам удобного веб-интерфейса для самостоятельной записи на услуги, что снизит нагрузку на администратора и исключит необходимость звонков и переписки в мессенджерах.

2. Создание визуально привлекательного представления услуг центра для формирования профессионального имиджа и привлечения новых клиентов.

3. Обеспечение встроенного интерфейса видеосвязи для проведения дистанционных психологических консультаций и диагностических приёмов непосредственно в приложении.

4. Организация интернет-магазина с корзиной и оформлением заказов для продажи товаров и видеокурсов онлайн.

5. Предоставление интерфейса для просмотра прямых трансляций с чатом для зрителей.

6. Обеспечение специалистов (психолог, специалист по чашам, администратор) удобными панелями управления для работы с расписанием, записями и контентом.

— страница записи на тибетские чаши с выбором города (Бийск/Новосибирск);

— журнал активности пользователей;
— управление инструкцией (guide) и настройками сайта.

1.3 Системные требования

Клиентская часть системы «Harmony Spa» представляет собой одностраничное приложение (SPA), работающее в браузере пользователя. Приложение является составной частью клиент-серверной архитектуры и взаимодействует с серверной частью исключительно через REST API.

Система состоит из следующих клиентских модулей:

1. Модуль маршрутизации (App.tsx) — определение маршрутов и связанных компонентов.
2. Модуль API (api.ts) — централизованный сервис взаимодействия с серверным REST API.
3. Модуль аутентификации — формы входа и регистрации с валидацией.
4. Модуль каталога и корзины — отображение товаров, управление корзиной, оформление заказов.
5. Модуль записи на услуги — страницы записи к психологу, на тибетские чаши (с выбором города) и на диагностику.
6. Модуль видеоконференцсвязи (VideoRoomPage.tsx) — WebRTC-клиент для видеосвязи с поддержкой демонстрации экрана.
7. Модуль трансляций — просмотр прямых эфиров с чатом, стриминг для администратора.
8. Модуль курсов — каталог, покупка и просмотр видеоуроков.
9. Модуль панелей управления — AdminPage, PsychologistPanel, BowlsSpecialistPanel.
10. Модуль общих компонентов — Header, LoadingScreen, формы.

Требования к среде выполнения клиентской части:
— современный веб-браузер: Google Chrome 90+, Mozilla Firefox 88+, Microsoft Edge 90+, Safari 14+;
— поддержка JavaScript ES2020, WebRTC API, MediaDevices API;
— поддержка CSS Grid, Flexbox;
— минимальное разрешение экрана: 320×568 px (мобильные устройства);
— активное интернет-соединение для взаимодействия с сервером.

Для сборки и разработки:
— Node.js 18+;
— npm (менеджер пакетов);
— Create React App (react-scripts) для сборки и dev-сервера.

1.4 Требования к пользовательскому интерфейсу

Визуальное оформление:
— Современный, минималистичный дизайн в спокойных тонах (мягкие оттенки, природная палитра), соответствующих тематике здоровья и красоты.
— Единообразный стиль всех страниц: общий хедер с навигацией, логотип, согласованная типографика.
— Адаптивная вёрстка: корректное отображение на мобильных устройствах (от 320 px), планшетах и десктопах.
— Анимированный загрузочный экран (LoadingScreen) при запуске приложения.

Навигация:
— Хедер с логотипом и навигационными ссылками, отображающий информацию об авторизации.
— Мобильное меню (бургер-меню) для узких экранов.
— Корзина в хедере с индикатором количества товаров.
— Клиентская маршрутизация без перезагрузки страницы (React Router).

Формы:
— Валидация на стороне клиента с отображением ошибок под полями.
— Обязательные поля помечены визуально.

Доступность:
— Семантическая HTML-разметка.
— Кнопки и интерактивные элементы достаточного размера для касания (мобильные устройства).
— Контрастный текст на фоне.

Видеоконференцсвязь:
— Полноэкранное отображение видео собеседника.
— Миниатюра собственного видео в углу экрана.
— Кнопки управления: включение/выключение камеры и микрофона, демонстрация экрана, завершение звонка.

2 АНАЛИЗ ТРЕБОВАНИЙ И ОПРЕДЕЛЕНИЕ СПЕЦИФИКАЦИЙ

2.1 Описание предметной области задачи ВКР

Предметная область данной работы — разработка пользовательского интерфейса для многопрофильного центра здоровья и красоты. Клиентская часть приложения отвечает за визуальное представление информации, взаимодействие с пользователем и передачу данных на сервер. С точки зрения фронтенд-разработки предметная область включает следующие направления:

— представление каталога товаров и услуг центра в удобном визуальном формате;
— интерактивные формы записи на приёмы с выбором параметров (дата, время, город, тип диагностики);
— встроенный интерфейс видеосвязи для дистанционных приёмов;
— интерфейс электронной коммерции (корзина, оформление заказа, история покупок);
— специализированные административные панели для различных ролей;
— мультимедийный контент: видеокурсы, прямые трансляции, аудио- и видеозаписи тибетских чаш.

Центр работает в двух городах — Новосибирске и Бийске, что отражается в интерфейсе записи на тибетский массаж: пользователь предварительно выбирает город, и расписание загружается с фильтрацией по выбранному городу.

2.1.1 Информационные объекты предметной области и взаимосвязи между ними

С точки зрения клиентского приложения информационные объекты представлены как типизированные интерфейсы TypeScript, используемые для обработки данных, полученных от REST API. Ключевые объекты:

1. Пользователь (User) — объект авторизованного пользователя: id, login, fullName, phone, isAdmin, isPsychologist, isBowlsSpecialist, createdAt. Хранится в состоянии React-компонента App и передаётся в дочерние компоненты.

2. Товар (Product) — объект каталога: id, name, description, price, oldPrice, category, image, courseId, partnerUrl. Отображается в компоненте ShopPage, фильтруется по категории.

3. Элемент корзины (CartItem) — пара (product, quantity). Отображается в CartPage, позволяет изменять количество и удалять позиции.

4. Заказ (Order) — id, total, createdAt, items[]. Отображается на странице истории покупок PurchaseHistoryPage.

5. Курс (Course) и видеоурок (CourseVideo) — объекты образовательного контента. Курс: id, title, description, price, coverImage, videos[]. Видеоурок: id, title, videoUrl, sortOrder.

6. Слот расписания (Slot) — элемент расписания специалиста: id, date, time, isBooked. Для тибетских чаш дополнительно содержит поле city. Отображается в компонентах записи (BookingPage, BowlsBookingPage, DiagnosticsBookingPage).

7. Запись на приём (Appointment) — id, date, time, status, roomId, userFullName, userPhone. Для тибетских чаш — дополнительно city. Для диагностики — roomId, roomActive, adminInRoom. Отображается в ProfilePage и панелях управления.

8. Трансляция (Stream) — id, title, description, date, time, speaker, status, price, isLive, streamRoomId, previewVideo. Отображается в StreamsPage.

9. Медиа тибетских чаш (BowlsMedia) — id, type (audio/video), title, fileUrl. Отображается в BowlsMediaPage с аудио- и видеоплеерами.

10. Запись журнала активности (ActivityLog) — id, userLogin, userFullName, action, details, createdAt. Отображается в панелях управления (AdminPage, PsychologistPanel, BowlsSpecialistPanel).

11. Инструкция (GuideItem) — id, title, content, sortOrder. Отображается в GuidePage.

Взаимосвязи между объектами на клиенте реализованы через идентификаторы: roomId связывает запись с видеокомнатой, courseId связывает товар с курсом, slotId связывает запись со слотом.

2.1.3 Методы работы с информационными объектами предметной области

2.1.3.1 Способы хранения информации об объектах предметной области

Клиентская часть не имеет собственного постоянного хранилища данных. Все данные хранятся на сервере и запрашиваются через REST API по мере необходимости. На стороне клиента используются следующие механизмы временного хранения:

1. Состояние React-компонентов (useState) — основной способ хранения данных на клиенте. Каждый компонент управляет собственным состоянием: списком товаров, содержимым корзины, выбранным слотом, текстом поиска и т.д. При размонтировании компонента состояние очищается.

2. Cookies (HTTP-only) — токен аутентификации хранится в HTTP-only cookie, устанавливаемом сервером. Клиентский JavaScript не имеет прямого доступа к токену; cookie автоматически прикрепляется браузером к каждому запросу через параметр credentials: 'include'.

3. Refs (useRef) — используются для хранения ссылок на объекты, не влияющие на рендеринг: RTCPeerConnection, MediaStream, HTML-элементы <video> и <audio>, идентификаторы интервалов polling.

4. URL-параметры (React Router) — параметры маршрута (/:roomId, /:streamId, /:courseId) используются для идентификации ресурса, отображаемого на странице.

Обоснование отсутствия localStorage/sessionStorage: все данные актуальны только в контексте текущей сессии авторизации; при выходе из системы данные должны быть недоступны. Использование серверного хранения обеспечивает целостность и актуальность информации.

2.1.3.2 Математические модели, используемые для обработки информации

На клиентской стороне применяются следующие алгоритмические решения:

1. Валидация форм — с использованием библиотеки Yup [9] определяются схемы валидации:
   — login: строка, обязательное поле, минимум 3 символа;
   — password: строка, обязательное поле, минимум 6 символов;
   — fullName: строка, обязательное поле;
   — phone: строка, обязательное поле.
   React Final Form [8] управляет состоянием формы и отображением ошибок валидации.

2. Фильтрация данных — клиентская фильтрация товаров по категории: products.filter(p => selectedCategory === 'all' || p.category === selectedCategory). Серверная фильтрация применяется для расписания по городу (параметр ?city=).

3. Группировка слотов по дате — для наглядного отображения расписания слоты группируются по полю date с использованием reduce:
   slots.reduce((acc, slot) => { (acc[slot.date] = acc[slot.date] || []).push(slot); return acc; }, {}).

4. Вычисление итоговой суммы корзины — cart.reduce((sum, item) => sum + item.price * item.quantity, 0).

5. Форматирование дат — преобразование ISO-дат (2026-06-15) в локальный формат отображения для русскоязычных пользователей.

2.1.3.5 Технологии получения и передачи информации

Клиентская часть использует следующие технологии:

1. Fetch API — встроенный браузерный API для выполнения HTTP-запросов. Все обращения к серверу проходят через централизованный модуль api.ts, который инкапсулирует вызовы fetch с параметрами credentials: 'include' (для передачи cookies) и headers Content-Type: application/json [5].

2. WebRTC API — браузерный API для организации видеоконференцсвязи [6]:
   — navigator.mediaDevices.getUserMedia() — захват камеры и микрофона;
   — navigator.mediaDevices.getDisplayMedia() — захват экрана для демонстрации;
   — RTCPeerConnection — установление P2P-соединения с собеседником;
   — SDP offer/answer и ICE candidates — обмен метаданными через сервер (long polling).

3. Long Polling — периодический опрос сервера (setInterval каждые 1–2 секунды) для получения новых WebRTC-сигналов и сообщений чата трансляции. Параметр lastId обеспечивает получение только новых данных.

4. HTML5 Media API — элементы <video> и <audio> для воспроизведения потоковых медиаданных:
   — установка srcObject для WebRTC-потоков;
   — установка src с URL сервера для воспроизведения файлов через HTTP Range.

5. FormData API — создание объектов FormData для загрузки файлов (изображения товаров, видеоуроки, аудио/видео чаш) через multipart/form-data.

2.1.4 Обзор существующих программных реализаций решения задачи

Для оценки степени новизны клиентской части рассмотрены пользовательские интерфейсы аналогичных решений:

1. Yclients [10] — веб-интерфейс записи в салоны красоты. Достоинства: удобный виджет онлайн-записи, встраиваемый в любой сайт; мобильное приложение. Недостатки: шаблонный дизайн без возможности глубокой кастомизации; отсутствие видеосвязи; нет интернет-магазина и видеокурсов; нет мультигородского выбора.

2. DIKIDI [11] — мобильно-ориентированный интерфейс записи. Достоинства: простой минималистичный дизайн; быстрая запись в несколько кликов. Недостатки: нет видеосвязи; нет e-commerce; нет трансляций; не поддерживает выбор города.

3. Bookly (WordPress) [12] — плагин с веб-интерфейсом записи. Достоинства: настраиваемый виджет; интеграция с WordPress-темами. Недостатки: привязан к WordPress; нет встроенной видеосвязи; ограниченный дизайн.

4. Zoom Client — десктопное/веб-приложение для видеосвязи. Достоинства: высококачественная видеосвязь; демонстрация экрана. Недостатки: отдельное приложение, не интегрированное с системой записи; требует регистрации; платные тарифы.

2.1.5 Концептуальное обоснование разработки

Оригинальный замысел клиентской части основан на следующих ключевых идеях:

1. Единое SPA-приложение, объединяющее функции записи на услуги, интернет-магазина, видеосвязи, просмотра курсов и трансляций, административного управления — без переключения между разными сайтами или приложениями.

2. Компонентная архитектура React с функциональными компонентами и хуками, обеспечивающая модульность, переиспользование кода и простоту поддержки.

3. Централизованный модуль API (api.ts, ~700 строк), инкапсулирующий все обращения к серверу в типизированные методы, что упрощает сопровождение и предотвращает дублирование кода.

4. Интеграция WebRTC непосредственно в SPA — пользователь переходит на страницу видеокомнаты по ссылке из личного кабинета, не покидая приложение. Реализована демонстрация экрана для удалённых консультаций.

5. Ролевая адаптация интерфейса — в зависимости от флагов пользователя (isAdmin, isPsychologist, isBowlsSpecialist) в личном кабинете отображаются ссылки на соответствующие панели управления; хедер адаптируется под роль.

2.2 Классы и характеристики пользователей

В клиентской части система различает 5 классов пользователей, интерфейс адаптируется под каждый:

1. Гость — неавторизованный пользователь. Видит главную страницу, каталог товаров, описания услуг, расписание трансляций. В хедере отображаются кнопки «Войти» и «Регистрация». Не может записываться, покупать, просматривать курсы. Характеристика: любой посетитель до входа в систему.

2. Клиент — авторизованный пользователь без специальных ролей. В хедере отображается имя пользователя и ссылка на личный кабинет. Доступны: запись на услуги, корзина, покупки, видеосвязь, курсы, трансляции. Характеристика: основная целевая аудитория.

3. Психолог (isPsychologist=1) — дополнительно к интерфейсу клиента имеет доступ к PsychologistPanel с вкладками: расписание, записи, пользователи, активность, инструкция. В личном кабинете отображается кнопка «Панель психолога».

4. Специалист по тибетским чашам (isBowlsSpecialist=1) — доступ к BowlsSpecialistPanel с вкладками: расписание (с выбором города), записи, пользователи, активность, инструкция. В личном кабинете — кнопка «Панель специалиста по тибетским чашам».

5. Администратор (isAdmin=1) — доступ к AdminPage с боковым меню и 15+ вкладками: товары, курсы, трансляции, пользователи, лаборатории, расписание диагностики, тибетские чаши, стриминг, активность, инструкция, настройки. В личном кабинете — кнопка «Панель администратора».

Роли комбинируются: пользователь может одновременно быть администратором и психологом; в личном кабинете будут отображены ссылки на обе панели.

2.3.2 Описание прецедентов

Прецедент 1: Регистрация и вход пользователя
Актор: Гость.
Предусловие: Пользователь находится на любой странице приложения.
Основной успешный сценарий:
1. Пользователь нажимает «Регистрация» в хедере.
2. Отображается форма с полями: логин, пароль, ФИО, телефон.
3. Пользователь заполняет поля и нажимает «Зарегистрироваться».
4. React Final Form выполняет валидацию через Yup-схему.
5. При успешной валидации вызывается API.user.register().
6. При успешном ответе сервера (200) пользователь перенаправляется на страницу входа.
7. Пользователь входит через форму авторизации; сервер устанавливает cookie.
8. Хедер обновляется: отображается имя пользователя и ссылка на личный кабинет.
Альтернативный сценарий: Ошибка валидации — под полем отображается текст ошибки на русском языке. Логин занят — отображается сообщение от сервера.

Прецедент 2: Запись на тибетский массаж с выбором города
Актор: Клиент (авторизован).
Предусловие: Пользователь перешёл на страницу записи на тибетские чаши.
Основной успешный сценарий:
1. Отображаются две кнопки выбора города: «Бийск» и «Новосибирск».
2. Отображается информационный блок о процедуре массажа «в четыре руки».
3. Пользователь нажимает «Новосибирск».
4. Компонент вызывает API.bowlsSchedule.getAvailable('novosibirsk').
5. Доступные слоты отображаются списком, сгруппированным по датам.
6. Пользователь выбирает слот, вводит ФИО и телефон.
7. Нажимает «Записаться».
8. Вызывается API.bowlsSchedule.book(slotId, fullName, phone).
9. Вызывается API.activity.log('Запись на тибетские чаши', 'Новосибирск, 2026-06-15 10:00').
10. Отображается сообщение об успешной записи.

Прецедент 3: Видеоконференция с демонстрацией экрана
Актор: Клиент (авторизован) и Психолог.
Предусловие: Существует запись со статусом in_progress и roomId.
Основной успешный сценарий:
1. Участник переходит по ссылке /video-room/:roomId из личного кабинета.
2. Компонент VideoRoomPage вызывает navigator.mediaDevices.getUserMedia({video: true, audio: true}).
3. Локальный видеопоток отображается в миниатюре.
4. Создаётся RTCPeerConnection с конфигурацией STUN/TURN-серверов.
5. Запускается polling: каждые 1-2 сек GET /room/:roomId/signals?lastId=N.
6. При получении SDP offer — создаётся answer и отправляется POST /room/:roomId/signal.
7. Обмениваются ICE-кандидатами.
8. Устанавливается P2P-соединение, видео собеседника отображается на полном экране.
9. Пользователь нажимает кнопку «Демонстрация экрана».
10. Вызывается navigator.mediaDevices.getDisplayMedia({video: true}).
11. Видеотрек в RTCPeerConnection заменяется на трек экрана через sender.replaceTrack().
12. Собеседник видит экран пользователя вместо камеры.
13. При повторном нажатии — переключение обратно на камеру.

Прецедент 4: Управление расписанием специалистом по чашам
Актор: Специалист по тибетским чашам.
Предусловие: Пользователь авторизован и имеет роль isBowlsSpecialist=1.
Основной успешный сценарий:
1. Пользователь переходит в «Панель специалиста по тибетским чашам».
2. Выбирает вкладку «Расписание».
3. Выбирает город из выпадающего списка (Бийск/Новосибирск).
4. Выбирает дату в поле ввода даты.
5. Отмечает чекбоксами нужные временные слоты.
6. Нажимает «Создать слоты».
7. Вызывается API.bowlsSchedule.create({date, times, city}).
8. Таблица расписания обновляется, новые слоты отображаются.

2.4 Нефункциональные требования

1. Производительность:
— Время загрузки главной страницы не более 3 секунд при скорости соединения 10 Мбит/с.
— Переключение между маршрутами (клиентская навигация) — мгновенно, без перезагрузки.
— Задержка в видеосвязи определяется качеством интернет-соединения; интерфейс не добавляет дополнительных задержек.

2. Совместимость:
— Chrome 90+, Firefox 88+, Edge 90+, Safari 14+.
— Адаптивный дизайн: мобильные устройства (от 320 px), планшеты, десктопы.
— Поддержка WebRTC во всех целевых браузерах.

3. Надёжность:
— Обработка ошибок API: все вызовы fetch обёрнуты в errorHandler, который парсит ответ сервера и выбрасывает ошибку с текстом на русском языке.
— При потере соединения отображается сообщение об ошибке.
— Polling для WebRTC-сигналов продолжает работать при кратковременных сбоях сети.

4. Безопасность:
— Токен аутентификации хранится в HTTP-only cookie, недоступной из JavaScript.
— Клиент не хранит пароли и конфиденциальные данные.
— Роли проверяются на сервере; клиент лишь адаптирует интерфейс.

5. Удобство развёртывания:
— npm install && npm start — запуск dev-сервера.
— npm run build — сборка production-версии в директорию build/.
— Собранное приложение представляет собой статические файлы (HTML, JS, CSS), которые могут быть размещены на любом веб-сервере.

6. Локализация:
— Весь интерфейс и сообщения на русском языке.
— Даты форматируются в локальном формате.

4.4 Реализация используемых методов хранения, обработки и передачи информации об объектах предметной области

4.4.1 Методы хранения данных на клиенте

Клиентская часть использует исключительно оперативное хранение данных в состоянии React-компонентов. Постоянное хранение (БД) делегировано серверной части.

Основные паттерны хранения:

1. Локальное состояние компонента (useState):
   const [products, setProducts] = useState<Product[]>([]);
   const [selectedCity, setSelectedCity] = useState<City | null>(null);
   const [error, setError] = useState('');
   При размонтировании компонента состояние автоматически очищается.

2. Рефы для мутабельных объектов (useRef):
   const pcRef = useRef<RTCPeerConnection | null>(null);
   const localStreamRef = useRef<MediaStream | null>(null);
   const remoteVideoRef = useRef<HTMLVideoElement | null>(null);
   const pollingRef = useRef<number | null>(null);
   Рефы не вызывают повторного рендеринга при изменении значения.

3. Ленивая загрузка данных (useEffect):
   useEffect(() => {
     API.products.getAll().then(setProducts);
   }, []);
   Данные загружаются при монтировании компонента и хранятся в состоянии до размонтирования.

4.4.5 Алгоритмы использования технологий передачи данных

Алгоритм взаимодействия с REST API (модуль api.ts):
1. Компонент вызывает типизированный метод: API.products.getAll().
2. Метод формирует URL: ${BASE_URL}/products.
3. Выполняется fetch с параметрами: method, headers (Content-Type: application/json), body (JSON.stringify(data)), credentials: 'include'.
4. Ответ проходит через errorHandler: если response.ok — возвращается response.json(); иначе парсится тело ответа и выбрасывается Error с полем message (текст на русском языке от сервера).
5. Компонент обрабатывает результат: обновляет состояние (setProducts) или отображает ошибку (setError).

Алгоритм WebRTC-соединения на клиенте (VideoRoomPage):
1. При монтировании компонента вызывается navigator.mediaDevices.getUserMedia({video: true, audio: true}).
2. Полученный MediaStream присваивается localStreamRef.current и отображается в <video ref={localVideoRef}> через srcObject.
3. Создаётся RTCPeerConnection с конфигурацией iceServers (STUN-сервера).
4. Все треки локального потока добавляются в PeerConnection: pc.addTrack(track, stream).
5. Устанавливается обработчик ontrack: при получении удалённого трека — он отображается в <video ref={remoteVideoRef}>.
6. Устанавливается обработчик onicecandidate: при генерации ICE-кандидата — отправляется POST /room/:roomId/signal.
7. Запускается polling через setInterval (1-2 сек):
   a. GET /room/:roomId/signals?lastId=N&excludeSender=userId.
   b. Для каждого полученного сигнала:
      — type='offer': pc.setRemoteDescription(offer), pc.createAnswer(), POST answer;
      — type='answer': pc.setRemoteDescription(answer);
      — type='candidate': pc.addIceCandidate(candidate).
8. При размонтировании компонента — clearInterval(polling), pc.close(), остановка треков.

Алгоритм демонстрации экрана:
1. Пользователь нажимает кнопку «Демонстрация экрана».
2. Вызывается navigator.mediaDevices.getDisplayMedia({video: true}).
3. Получается MediaStream экрана; screenStreamRef.current = screenStream.
4. Находится видео-sender в PeerConnection: const sender = pc.getSenders().find(s => s.track?.kind === 'video').
5. Видеотрек заменяется на трек экрана: sender.replaceTrack(screenStream.getVideoTracks()[0]).
6. Локальное видео в миниатюре обновляется.
7. Устанавливается обработчик screenStream.getVideoTracks()[0].onended — при закрытии окна демонстрации автоматически переключается обратно на камеру.
8. При повторном нажатии — трек заменяется обратно на трек камеры из localStreamRef.

Алгоритм polling чата трансляции:
1. При монтировании StreamViewerPage запускается setInterval (2 сек).
2. Каждые 2 секунды GET /stream-room/:roomId/messages?lastId=N.
3. Новые сообщения добавляются к массиву в состоянии: setMessages(prev => [...prev, ...newMessages]).
4. lastId обновляется до максимального id полученных сообщений.
5. При отправке сообщения пользователем — POST /stream-room/:roomId/messages.

4.5.2 Архитектура программного кода

Клиентский код организован по трёхуровневой архитектуре:

1. Слой данных (services/api.ts, ~700 строк):
Единый модуль, инкапсулирующий все обращения к серверному REST API. Экспортирует объект API с группами методов:
— API.auth: login(), logout()
— API.user: getUser(), register(), updateProfile()
— API.products: getAll(), create(), update(), delete()
— API.cart: get(), add(), remove(), updateQuantity(), checkout()
— API.courses: getAll(), getById(), create(), buy(), getMy()
— API.schedule: getSlots(), createSlots(), deleteSlot()
— API.appointments: book(), getMy(), start(), complete()
— API.bowlsSchedule: getAvailable(city), create(), delete(), book(), getSpecialistAppointments()
— API.diagnosticsSchedule: getSlots(), createSlots(), book(), activate(), close()
— API.room: sendSignal(), getSignals()
— API.streams: getAll(), create(), goLive(), endLive()
— API.streamRoom: sendSignal(), getSignals(), sendMessage(), getMessages()
— API.activity: log(), getLog(), getUsers()
— API.guide: getAll(), create(), update(), delete()
— API.labs, API.settings, API.bowlsMedia, API.admin

Каждый метод возвращает Promise с типизированным результатом.

2. Слой представления (views/, 30+ компонентов):
Функциональные React-компоненты, отвечающие за:
— отображение данных, полученных из слоя данных;
— обработку действий пользователя (клики, ввод, отправка форм);
— управление локальным состоянием (useState);
— управление побочными эффектами (useEffect — загрузка данных, polling);
— управление WebRTC-объектами (useRef для RTCPeerConnection, MediaStream).

Крупнейшие компоненты по объёму кода:
— AdminPage.tsx — ~1200 строк (15+ вкладок, CRUD для всех сущностей);
— Frame.tsx (HomePage) — ~1000 строк (уникальный дизайн лендинга);
— PsychologistPanel.tsx — ~600 строк;
— BowlsSpecialistPanel.tsx — ~350 строк;
— VideoRoomPage.tsx — ~300 строк (WebRTC + демонстрация экрана).

3. Слой общих компонентов (components/, 4 компонента):
— Header.tsx (~250 строк) — навигация, мобильное меню, корзина, авторизация;
— LoadingScreen.tsx — анимированный загрузочный экран;
— LoginForm — форма входа с React Final Form + Yup;
— RegistrationForm — форма регистрации с валидацией.

Паттерны взаимодействия между слоями:
— Компонент представления вызывает метод слоя данных → получает Promise → обновляет состояние → React перерисовывает DOM.
— Ошибки API перехватываются через catch и отображаются в UI через состояние error.
— Polling реализован через useEffect + setInterval + cleanup (clearInterval) при размонтировании.

Выполненные оптимизации:

1. Централизация API-вызовов — все обращения к серверу проходят через единый модуль api.ts, что исключает дублирование кода и упрощает обработку ошибок.

2. Очистка ресурсов при размонтировании — все useEffect с setInterval содержат cleanup-функцию (return () => clearInterval), предотвращающую утечки памяти. WebRTC-объекты (RTCPeerConnection, MediaStream) корректно закрываются.

3. Утилитарный CSS (Tailwind) — стили применяются непосредственно в JSX через классы, что исключает мёртвый CSS и упрощает поддержку. PostCSS purge удаляет неиспользуемые стили из production-сборки.

4. Условный рендеринг — компоненты панелей управления отображаются только при наличии соответствующей роли у пользователя, что исключает ненужные запросы к API.

5. Мемоизация колбэков — useCallback используется для функций, передаваемых как пропсы дочерним компонентам, предотвращая лишние перерендеры.
