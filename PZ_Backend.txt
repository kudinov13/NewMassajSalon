ПЕРЕЧЕНЬ СОКРАЩЕНИЙ, УСЛОВНЫХ ОБОЗНАЧЕНИЙ И ТЕРМИНОВ

API (Application Programming Interface) — интерфейс прикладного программирования, набор правил и спецификаций, позволяющих программным компонентам взаимодействовать друг с другом [1].

REST (Representational State Transfer) — архитектурный стиль взаимодействия компонентов распределённого приложения в сети, основанный на использовании стандартных методов протокола HTTP [2].

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня для передачи гипертекстовых документов, являющийся основой передачи данных в Интернете.

CRUD (Create, Read, Update, Delete) — четыре базовые операции работы с данными в информационных системах: создание, чтение, обновление и удаление.

SQLite — компактная встраиваемая реляционная СУБД, не требующая отдельного серверного процесса и хранящая всю базу данных в одном файле [3].

SQL (Structured Query Language) — язык структурированных запросов, предназначенный для управления данными в реляционных СУБД.

Node.js — серверная среда выполнения JavaScript на основе движка V8, использующая событийно-ориентированную неблокирующую модель ввода-вывода [1].

Express.js — минималистичный и гибкий веб-фреймворк для Node.js, предоставляющий средства маршрутизации и middleware для создания REST API [2].

WebRTC (Web Real-Time Communication) — технология, обеспечивающая передачу потоковых аудио- и видеоданных между браузерами в режиме реального времени без использования промежуточных серверов [4].

SDP (Session Description Protocol) — протокол описания параметров мультимедийной сессии, используемый при установлении WebRTC-соединения.

ICE (Interactive Connectivity Establishment) — фреймворк для установления соединения между двумя узлами в WebRTC, включающий механизмы обхода NAT [4].

Middleware — промежуточное программное обеспечение; в контексте Express.js — функция, выполняемая в цепочке обработки HTTP-запроса на сервере.

SPA (Single Page Application) — одностраничное приложение, в котором клиентская часть взаимодействует с сервером исключительно через API без полной перезагрузки страницы.

CORS (Cross-Origin Resource Sharing) — механизм, позволяющий веб-страницам выполнять запросы к ресурсам на другом домене, управляемый HTTP-заголовками.

Long Polling — техника периодического опроса сервера клиентом для получения новых данных в условиях отсутствия WebSocket-соединения.

Эндпоинт (endpoint) — конкретный URL-адрес на сервере, обрабатывающий определённый тип HTTP-запроса.

Сигнальный сервер — серверный компонент, обеспечивающий обмен метаданными (SDP, ICE) между участниками WebRTC-соединения для установления прямой P2P-связи.


ВВЕДЕНИЕ

В современном обществе наблюдается устойчивый рост интереса к альтернативным методам оздоровления, психологической поддержке и комплексному подходу к здоровью. Центры здоровья и красоты, предлагающие услуги психологического консультирования, тибетского массажа, диагностики и продажу натуральных продуктов, нуждаются в автоматизации бизнес-процессов для повышения эффективности работы и качества обслуживания клиентов.

Актуальность данной работы обусловлена тем, что большинство подобных центров до сих пор ведут запись клиентов вручную — через телефонные звонки и мессенджеры, не имеют единой системы учёта, а онлайн-консультации проводят через сторонние сервисы (Zoom, Skype), не интегрированные с системой записи. Это приводит к потере клиентов, путанице в расписании и невозможности масштабирования бизнеса.

Существующие программные решения в данной области, такие как Yclients [5], DIKIDI [6] и Bookly [7], ориентированы на типовые салоны красоты и парикмахерские. Они не поддерживают специфический функционал, необходимый для многопрофильного центра здоровья: видеоконсультации для психологических сеансов и диагностики, запись на массаж тибетскими чашами с выбором города, продажу видеокурсов, прямые трансляции мастер-классов и систему отслеживания активности пользователей. Указанные недостатки обуславливают необходимость создания собственного программного продукта.

Целью данной выпускной квалификационной работы является разработка серверной части (бэкенда) веб-приложения «Harmony Spa», обеспечивающей управление базой данных, аутентификацию и авторизацию пользователей с ролевой моделью доступа, REST API для всех функциональных модулей системы, сигнальный сервер для организации видеоконференцсвязи и потоковую раздачу медиафайлов.

Для достижения поставленной цели определены следующие задачи:
— провести анализ предметной области и определить требования к серверной части;
— спроектировать реляционную базу данных с учётом всех информационных объектов предметной области;
— реализовать REST API на платформе Node.js с использованием фреймворка Express.js;
— разработать систему аутентификации на основе токенов с использованием HTTP-only cookies;
— реализовать сигнальный сервер для WebRTC видеоконференций и прямых трансляций;
— реализовать потоковую раздачу видео- и аудиофайлов с поддержкой HTTP Range;
— провести тестирование серверной части.

Практическая значимость работы заключается в создании полностью функциональной серверной инфраструктуры, пригодной для развёртывания и использования реальным центром здоровья и красоты.


1 ПОСТАНОВКА ЗАДАЧИ ВКР

1.1 Бизнес-требования

Заказчиком разработки является центр здоровья и красоты «Harmony Spa», осуществляющий деятельность в городах Новосибирск и Бийск. Центр предоставляет широкий спектр услуг: психологическое консультирование, массаж тибетскими чашами (уникальная процедура «в четыре руки»), визуальную диагностику состояния здоровья, продажу натуральных продуктов и образовательных видеокурсов, проведение прямых трансляций мастер-классов.

Основные бизнес-цели, которые организация намерена достичь с помощью разрабатываемой системы:

1. Автоматизация записи клиентов на приём к психологу, на сеансы тибетского массажа и диагностику, что позволит исключить человеческий фактор и снизить нагрузку на администратора.

2. Организация онлайн-консультаций — видеоконференции для психологических сеансов и диагностических приёмов должны быть встроены непосредственно в систему записи, без использования сторонних сервисов.

3. Электронная коммерция — продажа товаров (натуральная косметика, БАДы) и видеокурсов через интернет-магазин с корзиной и историей покупок.

4. Проведение прямых трансляций — мастер-классы и лекции от специалистов центра с чатом для зрителей в реальном времени.

5. Мониторинг активности — отслеживание действий пользователей на сайте (записи, покупки, регистрации) для аналитики и контроля качества обслуживания.

6. Мультигородская работа — возможность ведения расписания отдельно для каждого города (Бийск и Новосибирск), что соответствует географии деятельности центра.

1.2 Пользовательские требования

Серверная часть должна обеспечивать следующие возможности для конечных пользователей системы:

Для клиентов (обычные пользователи):
— регистрация и авторизация с сохранением сессии между посещениями;
— просмотр каталога товаров с фильтрацией по категориям;
— работа с корзиной: добавление товаров, изменение количества, удаление, оформление заказа;
— запись на приём к психологу с выбором даты и времени из доступного расписания;
— запись на сеанс тибетского массажа с предварительным выбором города;
— запись на диагностику (5 типов: ногти, язык, глаза, кожа, тело);
— участие в видеоконференциях с психологом и специалистом по диагностике;
— просмотр приобретённых видеокурсов;
— просмотр прямых трансляций и участие в чате;
— просмотр истории покупок и записей в личном кабинете.

Для психолога:
— управление расписанием (создание и удаление слотов приёма);
— просмотр списка записей клиентов с контактными данными;
— инициирование и завершение видеоконференции с клиентом;
— поиск пользователей по имени, логину или телефону;
— просмотр журнала активности пользователей.

Для специалиста по тибетским чашам:
— управление расписанием с обязательной привязкой слотов к городу (Бийск или Новосибирск);
— просмотр записей клиентов с указанием города;
— загрузка аудио- и видеоматериалов звучания чаш;
— поиск пользователей и просмотр журнала активности.

Для администратора:
— полный доступ ко всем функциям системы;
— управление товарами, курсами, трансляциями, лабораториями;
— управление пользователями и назначение ролей;
— управление расписанием диагностики;
— ведение прямых эфиров;
— просмотр полного журнала активности и аналитика.

1.3 Системные требования

Серверная часть системы «Harmony Spa» представляет собой REST API сервер, являющийся ключевым компонентом клиент-серверной архитектуры. Разрабатываемое программное средство является частью более крупной системы, включающей также клиентское React-приложение (SPA). Серверная часть играет роль единого центра обработки данных и бизнес-логики.

Система состоит из следующих подсистем (модулей), выделенных по функциональному признаку:

1. Модуль аутентификации и авторизации — регистрация пользователей, проверка учётных данных, выдача и хранение токенов, ролевое разграничение доступа.
2. Модуль каталога товаров — CRUD-операции для товаров, фильтрация по категориям, загрузка изображений.
3. Модуль корзины и заказов — управление корзиной, оформление заказа, история покупок.
4. Модуль курсов — управление видеокурсами, загрузка видеоуроков, предоставление доступа.
5. Модуль расписания и записей — три подмодуля: расписание психолога, расписание тибетских чаш (с привязкой к городу), расписание диагностики.
6. Модуль видеоконференцсвязи — сигнальный сервер для обмена WebRTC-метаданными.
7. Модуль трансляций — управление прямыми эфирами, WebRTC-сигнализация для стриминга, чат.
8. Модуль медиа-контента — загрузка и потоковая раздача аудио- и видеофайлов.
9. Модуль активности — журналирование действий пользователей в системе.
10. Модуль настроек и контента — управление настройками сайта, инструкцией, лабораториями.

Требования к среде выполнения серверной части:
— Node.js версии 18 или выше;
— операционная система: Windows 10+, Linux (Ubuntu 20.04+) или macOS 12+;
— минимум 512 МБ оперативной памяти;
— наличие файловой системы для хранения БД (файл database.db) и загруженных файлов (директория uploads/).

1.4 Требования к пользовательскому интерфейсу

Серверная часть не имеет собственного графического пользовательского интерфейса. Взаимодействие с ней осуществляется исключительно посредством REST API по протоколу HTTP. Требования к API как интерфейсу взаимодействия:

Формат данных:
— Все запросы с телом передаются в формате JSON (Content-Type: application/json).
— Все ответы сервера представлены в формате JSON.
— Загрузка файлов осуществляется через multipart/form-data.

Коды ответов:
— 200 — успешное выполнение запроса;
— 400 — ошибка валидации входных данных;
— 401 — пользователь не авторизован;
— 403 — доступ запрещён (недостаточно прав);
— 404 — ресурс не найден;
— 500 — внутренняя ошибка сервера.

Сообщения об ошибках возвращаются на русском языке в поле «message» JSON-ответа, что обеспечивает возможность отображения их пользователю без дополнительной локализации на клиенте.

Аутентификация осуществляется через HTTP-only cookies с флагами SameSite и Secure, что предотвращает доступ к токену из клиентского JavaScript и защищает от CSRF-атак.

API поддерживает CORS (Cross-Origin Resource Sharing) для обеспечения возможности работы клиентской части, развёрнутой на другом порту или домене.

2 АНАЛИЗ ТРЕБОВАНИЙ И ОПРЕДЕЛЕНИЕ СПЕЦИФИКАЦИЙ

2.1 Описание предметной области задачи ВКР

Предметная область данной работы — автоматизация деятельности многопрофильного центра здоровья и красоты, предоставляющего широкий спектр услуг: психологическое консультирование, массаж тибетскими чашами, визуальная диагностика состояния здоровья по различным признакам (ногти, язык, глаза, кожа, тело), продажа натуральных продуктов (косметика, БАДы) и образовательных видеокурсов, а также проведение прямых трансляций мастер-классов и лекций.

Центр здоровья «Harmony Spa» работает в двух городах — Новосибирске и Бийске, что требует раздельного ведения расписания для каждого филиала. Приём психолога и диагностика проводятся дистанционно по видеосвязи, что обуславливает необходимость встроенной системы видеоконференцсвязи в рамках платформы. Массаж тибетскими чашами проводится очно и представляет собой уникальную процедуру «в четыре руки», когда два мастера одновременно работают с телом клиента.

Ключевые проблемы предметной области, требующие программного решения:
— отсутствие единой автоматизированной системы управления записями, что приводит к конфликтам в расписании и потере клиентов;
— необходимость встроенной видеосвязи вместо использования разрозненных сторонних сервисов, не связанных с системой записи;
— потребность в мониторинге действий пользователей для аналитики и улучшения качества обслуживания;
— отсутствие единой платформы для продажи товаров и видеокурсов;
— необходимость мультигородского управления расписанием при работе в нескольких филиалах.

2.1.1 Информационные объекты предметной области и взаимосвязи между ними

Основные информационные объекты системы «Harmony Spa»:

1. Пользователь (User) — центральный информационный объект системы. Содержит учётные данные (логин, хеш пароля), персональные данные (ФИО, телефон), набор флагов ролей (администратор, психолог, специалист по тибетским чашам) и дату создания аккаунта. Каждый пользователь однозначно идентифицируется целочисленным автоинкрементным первичным ключом.

2. Токен аутентификации (Token) — объект, связывающий сессию пользователя с его учётной записью. Каждый токен содержит уникальную строку (UUID) и ссылку на идентификатор пользователя. Связь с пользователем: многие-к-одному (один пользователь может иметь несколько активных сессий).

3. Товар (Product) — объект каталога интернет-магазина. Содержит название, описание, текущую цену, старую цену (для отображения скидки), категорию, путь к изображению, опциональную связь с курсом и ссылку на товар у партнёра (для категории «БАДы»).

4. Корзина (Cart) — связующий объект между пользователем и товаром с указанием количества. Уникальность обеспечивается парой (userId, productId).

5. Заказ (Order) и позиция заказа (OrderItem) — фиксация оформленных покупок. Заказ содержит ссылку на пользователя и итоговую сумму. Позиции заказа хранят название товара, цену и количество на момент покупки (денормализация для сохранения истории). Связь: один заказ — много позиций.

6. Курс (Course) и видеоурок (CourseVideo) — образовательный контент. Курс содержит название, описание, цену и обложку. Видеоуроки связаны с курсом через внешний ключ с каскадным удалением и имеют поле порядка сортировки. Связь: один курс — много видеоуроков.

7. Покупка курса (UserCourse) — связь пользователя с приобретённым курсом. Уникальность: пара (userId, courseId).

8. Слот расписания (ScheduleSlot) — элемент расписания специалиста. Существует три разновидности: слот психолога (schedule_slots), слот тибетских чаш (bowls_schedule_slots, дополнительно содержит поле города) и слот диагностики (diagnostics_schedule_slots). Каждый слот содержит дату, время, ссылку на специалиста и признак бронирования.

9. Запись на приём (Appointment) — связь клиента и специалиста через забронированный слот. Для психолога (appointments) содержит roomId — уникальный идентификатор комнаты видеосвязи, статус (booked, in_progress, completed), ФИО и телефон клиента. Для тибетских чаш (bowls_appointments) дополнительно содержит город. Для диагностики (diagnostics_appointments) — roomId и флаги состояния комнаты.

10. Сигнал видеокомнаты (RoomSignal) — временный объект, хранящий WebRTC-метаданные (SDP offer, answer, ICE candidates) для установления P2P-соединения между участниками видеоконференции. Удаляется после завершения сеанса.

11. Трансляция (Stream) — информационный объект прямого эфира. Содержит название, описание, дату, время, имя спикера, статус (planned, live, finished), цену (0 для бесплатных), идентификатор комнаты стриминга и путь к превью-видео.

12. Подписка на трансляцию (UserStream) — связь пользователя с трансляцией (регистрация или покупка). Уникальность: пара (userId, streamId).

13. Сообщение чата трансляции (StreamMessage) — текстовое сообщение от пользователя в чате прямого эфира.

14. Журнал активности (UserActivityLog) — запись действия пользователя с отметкой времени. Содержит ссылку на пользователя, его логин и ФИО (денормализация для удобства поиска), описание действия и дополнительные детали.

15. Настройки (Settings) — хранилище параметров сайта в формате ключ-значение (например, адрес центра).

16. Лаборатория (Lab) — партнёрская организация для сдачи анализов. Содержит название, организацию и URL.

17. Медиа тибетских чаш (BowlsMedia) — аудио- и видеозаписи звучания тибетских чаш с указанием типа (audio/video), названия и пути к файлу.

18. Инструкция (GuideItem) — пункт навигационной инструкции для пользователей сайта с заголовком, текстом и порядком сортировки.

Основные взаимосвязи между объектами:
— Пользователь является центральным объектом и связан через внешние ключи с корзиной, заказами, записями на приём, курсами, трансляциями и журналом активности.
— Слоты расписания связаны с записями (один слот — одна запись).
— Курсы связаны с видеоуроками (один ко многим с каскадным удалением).
— Товары опционально связаны с курсами (при покупке товара-курса предоставляется доступ к видеоурокам).
— Трансляции связаны с подписками, сообщениями чата, зрителями и сигналами стриминга.

2.1.2 Информационные потребности пользователей

Клиенты системы получают информацию в следующем виде:
— каталог товаров с ценами, описаниями и изображениями, с возможностью фильтрации по категориям;
— расписание доступных слотов для записи на приём (психолог), тибетский массаж (с разделением по городам) и диагностику;
— статус своих записей (забронировано, в процессе, завершено) и возможность входа в видеоконференцию при наступлении времени приёма;
— список приобретённых видеокурсов с доступом к видеоурокам;
— история оформленных заказов с детализацией по позициям;
— расписание предстоящих трансляций с информацией о спикере, цене и статусе;
— аудио- и видеозаписи звучания тибетских чаш для предварительного ознакомления.

Специалисты (психолог и специалист по чашам) получают:
— список записей клиентов с контактными данными (ФИО, телефон);
— состояние расписания с информацией о занятости слотов;
— возможность поиска пользователей по имени, логину или телефону;
— журнал активности пользователей — кто, когда и какие действия совершал на сайте;
— роли-специфическую инструкцию по работе с панелью управления.

Администратор получает:
— полную статистику системы: количество пользователей, товаров, курсов, трансляций;
— журнал всех действий пользователей с возможностью поиска и фильтрации;
— возможность управления ролями пользователей (назначение/снятие прав);
— возможность управления всем контентом сайта через административную панель.

2.1.3 Методы работы с информационными объектами предметной области

2.1.3.1 Способы хранения информации об объектах предметной области

Для хранения структурированных данных системы выбрана реляционная СУБД SQLite. Обоснование выбора:
— не требует отдельного серверного процесса (встраиваемая архитектура);
— вся база данных хранится в одном файле (database.db), что упрощает резервное копирование и переносимость;
— полная поддержка стандартного SQL, внешних ключей и транзакций;
— идеально подходит для приложений с одним сервером и умеренной нагрузкой (до нескольких сотен одновременных пользователей);
— нулевая конфигурация при развёртывании — достаточно запустить приложение.

Структура хранения разделена на два уровня:
1. Реляционные данные (пользователи, товары, заказы, расписание, записи, журнал активности и пр.) хранятся в таблицах SQLite. База данных содержит 22 таблицы, описание которых приведено в разделе 4.4.1.
2. Бинарные медиафайлы (изображения товаров, видеоуроки курсов, аудио- и видеозаписи тибетских чаш, превью трансляций) хранятся в файловой системе сервера в директории /uploads. Пути к файлам сохраняются в соответствующих полях таблиц БД как текстовые значения.

Такое разделение обосновано тем, что хранение бинарных данных непосредственно в SQLite неэффективно для файлов размером более 1 МБ и затрудняет потоковую раздачу медиа-контента.

2.1.3.2 Математические модели, используемые для обработки информации

В данной работе не применяются сложные математические модели. Тем не менее, используются следующие алгоритмические и математические решения:

1. Хеширование паролей — для безопасного хранения паролей применяется алгоритм MD5. При регистрации пользователя пароль преобразуется в 128-битный хеш: h = MD5(password), который сохраняется в базе данных. При авторизации введённый пароль хешируется аналогичным образом и сравнивается с сохранённым хешем [8].

2. Генерация уникальных идентификаторов — для создания токенов аутентификации и идентификаторов видеокомнат используется функция crypto.randomUUID(), генерирующая UUID версии 4 на основе криптографически стойкого генератора случайных чисел.

3. Правило ограничения записи по времени — для предотвращения записи на слот, до которого осталось менее 2 часов, выполняется вычисление:
   delta = (slotHour * 60 + slotMinutes) - (currentHour * 60 + currentMinutes)
   Запись разрешена если delta >= 120 минут (для слотов на текущий день).

4. Агрегация данных — подсчёт итоговой суммы заказа выполняется SQL-агрегацией: SELECT SUM(price * quantity) FROM cart_items WHERE userId = ?.

2.1.3.5 Технологии получения и передачи информации

Серверная часть использует следующие технологии передачи данных:

1. REST API через HTTP — основной способ взаимодействия клиента с сервером. Клиентское приложение отправляет HTTP-запросы (GET, POST, PUT, DELETE) с данными в формате JSON, сервер обрабатывает запрос и возвращает ответ в JSON. Базовый URL сервера: http://localhost:3001 [2].

2. HTTP-only Cookies — механизм передачи токена аутентификации. При успешной авторизации сервер устанавливает cookie с параметрами: httpOnly (недоступен из JavaScript), SameSite, Secure (в production). Cookie автоматически прикрепляется браузером к каждому последующему запросу.

3. HTTP Range Requests — технология потоковой передачи видео- и аудиофайлов [9]. Клиент запрашивает фрагмент файла через заголовок Range (например, Range: bytes=0-1048575), сервер отвечает кодом 206 Partial Content с запрошенным фрагментом. Это обеспечивает перемотку видео и экономию трафика.

4. Long Polling — техника получения WebRTC-сигналов. Клиенты периодически опрашивают сервер (GET /room/:roomId/signals?lastId=N) для получения новых SDP-офферов, ответов и ICE-кандидатов. Параметр lastId обеспечивает получение только новых сигналов.

5. WebRTC сигнализация — сервер выступает посредником при установлении P2P-соединения между участниками видеоконференции: принимает, хранит и передаёт SDP (Session Description Protocol) офферы, ответы и ICE-кандидаты [4].

2.1.4 Обзор существующих программных реализаций решения задачи

Для оценки степени новизны разрабатываемого программного средства рассмотрены следующие частичные и полные аналоги:

1. Yclients [5] — популярная российская платформа для онлайн-записи в салоны красоты и медицинские центры. Достоинства: интеграция с мессенджерами и социальными сетями, мобильное приложение, автоматические напоминания. Недостатки: отсутствие встроенной видеоконференцсвязи для дистанционных приёмов; отсутствие поддержки мультигородского расписания; невозможность продажи видеокурсов и проведения прямых трансляций; платная подписка от 1500 руб./мес; невозможность кастомизации под специфику тибетского массажа.

2. DIKIDI [6] — платформа для записи в салоны красоты и фитнес-центры. Достоинства: бесплатный базовый тариф, мобильное приложение, интеграция с Instagram. Недостатки: отсутствие видеосвязи; отсутствие электронной коммерции (продажа товаров, курсов); невозможность проведения трансляций; ограниченные возможности мониторинга активности; невозможность привязки расписания к разным городам.

3. Bookly (WordPress-плагин) [7] — система онлайн-записи, интегрируемая в сайт на WordPress. Достоинства: гибкая настройка, интеграция с платёжными системами, открытый API. Недостатки: привязка к экосистеме WordPress; отсутствие встроенной видеосвязи; невозможность продажи курсов и товаров через единый интерфейс; сложная настройка для нетиповых услуг.

4. Zoom / Google Meet — сервисы видеоконференцсвязи. Достоинства: высокое качество связи, стабильность, кросс-платформенность. Недостатки: не интегрированы с системами онлайн-записи; требуют отдельной регистрации клиента; невозможно автоматизировать создание комнат из системы записи; дополнительные расходы на лицензии.

2.1.5 Концептуальное обоснование разработки

Концепция разработки основана на создании единой серверной платформы, объединяющей все бизнес-процессы многопрофильного центра здоровья в одном REST API. Оригинальный замысел решения проблемы заключается в следующих ключевых идеях:

1. Интеграция сигнального сервера для WebRTC непосредственно в основной Express-сервер приложения. Это исключает необходимость сторонних сервисов видеосвязи и позволяет автоматически создавать видеокомнаты при бронировании приёма.

2. Использование единой базы данных для всех функциональных модулей, что обеспечивает целостность данных и позволяет связывать информацию из разных подсистем (например, покупка товара-курса автоматически предоставляет доступ к видеоурокам).

3. Гибкая ролевая модель, при которой каждый пользователь может одновременно совмещать несколько ролей (например, быть одновременно администратором и психологом), что реализовано через независимые булевые флаги вместо единственного поля роли.

4. Мультигородская система расписания через единый API с параметрической фильтрацией, что позволяет масштабировать систему на произвольное количество городов без изменения архитектуры.

2.2 Классы и характеристики пользователей

В системе «Harmony Spa» выделены следующие пользовательские роли (классы пользователей):

1. Гость — неавторизованный пользователь. Может просматривать каталог товаров, описания услуг, расписание трансляций. Не имеет доступа к записи на приём, покупкам и личному кабинету. Характеристика: любой посетитель сайта до регистрации или входа в систему.

2. Клиент (обычный пользователь) — авторизованный пользователь с флагами isAdmin=0, isPsychologist=0, isBowlsSpecialist=0. Может совершать покупки, записываться на приёмы, участвовать в видеоконференциях, просматривать курсы и трансляции. Характеристика: основная аудитория сайта — клиенты центра здоровья.

3. Психолог — пользователь с флагом isPsychologist=1. Дополнительно к функциям клиента имеет доступ к панели управления расписанием, может видеть записи клиентов, инициировать видеоконференции, искать пользователей и просматривать журнал активности. Характеристика: сотрудник центра, проводящий онлайн-консультации.

4. Специалист по тибетским чашам — пользователь с флагом isBowlsSpecialist=1. Дополнительно может управлять расписанием с привязкой к городу, видеть записи клиентов, загружать аудио- и видеоматериалы тибетских чаш. Характеристика: мастер тибетского массажа, работающий в одном или нескольких городах.

5. Администратор — пользователь с флагом isAdmin=1. Имеет полный доступ ко всем функциям системы: управление товарами, курсами, пользователями, трансляциями, расписанием диагностики, ведение прямых эфиров, просмотр журнала активности. Характеристика: управляющий центром или ответственный сотрудник.

Важная особенность: роли не являются взаимоисключающими. Один пользователь может одновременно обладать несколькими ролями (например, быть администратором и психологом). Это реализовано через независимые целочисленные флаги в таблице users.

2.3.2 Описание прецедентов

Прецедент 1: Регистрация пользователя
Актор: Гость.
Предусловие: Пользователь не авторизован.
Основной успешный сценарий:
1. Клиент отправляет POST /user с данными {login, password, fullName, phone}.
2. Сервер проверяет, что пользователь с таким логином не существует.
3. Сервер вычисляет MD5-хеш пароля и создаёт нового пользователя.
4. Сервер записывает в журнал активности действие «Регистрация».
5. Сервер возвращает 200 с данными нового пользователя.
Альтернативный сценарий: Логин занят — сервер возвращает 400 с сообщением.

Прецедент 2: Бронирование приёма к психологу
Актор: Клиент (авторизован).
Предусловие: Существуют свободные слоты в расписании психолога.
Основной успешный сценарий:
1. Клиент отправляет POST /appointments с {slotId, fullName, phone}.
2. Сервер проверяет, что слот существует и не забронирован.
3. Сервер проверяет правило 2 часов (для слотов на сегодня: delta >= 120 мин).
4. Сервер генерирует уникальный roomId (UUID) для видеоконференции.
5. Сервер отмечает слот как занятый (isBooked = 1).
6. Сервер создаёт запись в таблице appointments.
7. Сервер обновляет профиль пользователя (ФИО, телефон).
8. Сервер записывает действие в журнал активности.
9. Сервер возвращает 200 с данными записи, включая roomId.
Альтернативный сценарий: Слот уже занят — возвращается 400.

Прецедент 3: Проведение видеоконференции
Акторы: Психолог и Клиент.
Предусловие: Существует запись со статусом booked.
Основной успешный сценарий:
1. Психолог отправляет POST /appointments/:id/start.
2. Сервер обновляет статус записи на in_progress, очищает старые сигналы.
3. Психолог создаёт RTCPeerConnection, генерирует SDP offer.
4. Психолог отправляет POST /room/:roomId/signal с {type: 'offer', data: sdp}.
5. Сервер сохраняет сигнал в таблице room_signals.
6. Клиент опрашивает GET /room/:roomId/signals?lastId=N, получает offer.
7. Клиент создаёт SDP answer и отправляет POST /room/:roomId/signal.
8. Оба участника обмениваются ICE-кандидатами аналогичным образом.
9. Устанавливается прямое P2P-соединение.
10. По завершении психолог отправляет POST /appointments/:id/complete.
11. Сервер обновляет статус на completed.

Прецедент 4: Запись на тибетский массаж с выбором города
Актор: Клиент (авторизован).
Предусловие: Специалист создал слоты с привязкой к городу.
Основной успешный сценарий:
1. Клиент запрашивает GET /bowls-schedule?city=novosibirsk.
2. Сервер возвращает доступные слоты для выбранного города.
3. Клиент выбирает слот и отправляет POST /bowls-schedule/book с {slotId, fullName, phone}.
4. Сервер проверяет слот, отмечает как занятый, создаёт запись с указанием города.
5. Сервер записывает действие в журнал активности.
6. Сервер возвращает 200 с данными записи.

Прецедент 5: Проведение прямой трансляции
Актор: Администратор.
Предусловие: Создана трансляция со статусом planned.
Основной успешный сценарий:
1. Администратор отправляет POST /streams/:id/go-live.
2. Сервер обновляет статус на live, устанавливает isLive = 1, генерирует streamRoomId.
3. Администратор начинает вещание, отправляя WebRTC-сигналы через POST /stream-room/:roomId/signal.
4. Зрители подключаются, получают сигналы через GET /stream-room/:roomId/signals.
5. Участники обмениваются текстовыми сообщениями через POST/GET /stream-room/:roomId/messages.
6. По окончании администратор отправляет POST /streams/:id/end-live.

2.4 Нефункциональные требования

Нефункциональные требования определяют условия и среду выполнения функций серверной части системы.

1. Производительность и масштабируемость:
— Время ответа сервера на стандартный REST-запрос не должно превышать 200 мс при нормальной нагрузке.
— SQLite обеспечивает до 100 000 операций INSERT в секунду, что достаточно для целевой аудитории.
— Потоковая раздача видео через HTTP Range позволяет обслуживать нескольких пользователей одновременно без загрузки полных файлов в оперативную память.

2. Переносимость и совместимость:
— Серверная часть работает на Node.js 18+ под управлением Windows, Linux и macOS.
— REST API совместим с любым HTTP-клиентом (браузер, мобильное приложение, Postman).
— CORS настроен для работы с клиентской частью на любом порту и домене.
— SQLite не требует установки отдельного сервера СУБД.

3. Надёжность и доступность:
— Безопасные миграции базы данных: все ALTER TABLE обёрнуты в блоки try-catch, что обеспечивает корректный запуск при любом состоянии БД (идемпотентность).
— Автоматическое создание всех необходимых таблиц и тестовых данных при первом запуске.
— Файл базы данных сохраняется между перезапусками сервера.
— Логирование активности выполняется в отдельном блоке try-catch и не прерывает основной поток обработки запросов при сбоях.

4. Безопасность:
— Пароли хранятся исключительно в виде MD5-хешей, исходные пароли не сохраняются.
— Токены аутентификации передаются в HTTP-only cookies, недоступных из клиентского JavaScript.
— Разграничение доступа реализовано на уровне каждого эндпоинта через middleware (requireAuth, requireAdmin, requirePsychologist, requireBowls).
— Cookie установлены с флагами SameSite и Secure в production-окружении.

5. Удобство развёртывания:
— Минимальная конфигурация: npm install && node index.js.
— Нулевая конфигурация СУБД (SQLite создаёт файл database.db автоматически).
— Автоматическое создание тестовых учётных записей при первом запуске.

6. Локализация:
— Все сообщения об ошибках и системные тексты возвращаются на русском языке.
— Начальные данные инструкции (guide) заполнены на русском языке.

3.2 Характеристика выбранных программных сред и средств

Node.js v18+ — серверная среда выполнения JavaScript, построенная на движке V8 от Google. Использует событийно-ориентированную неблокирующую модель ввода-вывода (event loop), что обеспечивает высокую производительность при обработке большого числа одновременных соединений. Встроенные модули crypto, fs, path используются в проекте для генерации UUID-токенов, работы с файловой системой и формирования путей [1].

Express.js v4.x — минималистичный и гибкий веб-фреймворк для Node.js. Предоставляет механизм маршрутизации (Router), цепочки middleware для обработки запросов, поддержку статических файлов. В проекте используется для определения REST API эндпоинтов, обработки JSON-тел запросов, работы с cookies и управления CORS [2].

sqlite3 + sqlite (npm-пакеты) — библиотека sqlite3 предоставляет нативную привязку к SQLite на C++ через N-API. Пакет sqlite добавляет обёртку с поддержкой Promise и async/await. Используемые методы: db.run() для модифицирующих запросов, db.get() для получения одной записи, db.all() для получения массива записей, db.exec() для выполнения DDL-команд [3].

md5 (npm-пакет) — библиотека для вычисления MD5-хеша строк. Используется для хеширования паролей при регистрации и авторизации.

cookie-parser (npm-пакет) — middleware для Express, обеспечивающий автоматический парсинг cookies из заголовков HTTP-запросов. Извлечённые cookies доступны через объект req.cookies.

cors (npm-пакет) — middleware для настройки CORS-заголовков. Настроен с параметрами credentials: true (для передачи cookies) и origin: true (автоматическая подстановка Origin).

multer (npm-пакет) — middleware для обработки загрузки файлов через multipart/form-data. Используется для загрузки изображений товаров, видеоуроков курсов, аудио- и видеозаписей тибетских чаш и превью-видео трансляций. Настроен на сохранение файлов в директорию uploads/ с генерацией уникальных имён.

crypto (встроенный модуль Node.js) — используется метод crypto.randomUUID() для генерации уникальных токенов аутентификации и идентификаторов видеокомнат.

fs, path (встроенные модули Node.js) — используются для проверки существования файлов (fs.existsSync), получения размера файла (fs.statSync), создания потоков чтения (fs.createReadStream) и формирования путей к файлам (path.join, path.extname).

Visual Studio Code — среда разработки с поддержкой отладки Node.js, IntelliSense для JavaScript и интегрированным терминалом.

4.1 Этапы реализации

Реализация серверной части выполнена в следующей последовательности этапов:

1. Инициализация проекта — создание структуры каталогов (server/, server/db/, server/routes/, server/uploads/), настройка package.json, установка зависимостей (express, sqlite3, sqlite, md5, cookie-parser, cors, multer).

2. Проектирование и реализация базы данных — определение схемы 22 таблиц, написание функции initDb() с автоматическим созданием таблиц, миграциями для существующих таблиц и начальными данными (тестовые пользователи, инструкция, трансляция).

3. Реализация слоя доступа к данным — создание модулей в директории server/db/ для работы с пользователями (users.js), токенами (tokens.js), товарами (products.js), корзиной (cart.js), трансляциями (streams.js), настройками (settings.js), лабораториями (labs.js), журналом активности (activity.js).

4. Реализация middleware авторизации — функции extractAuth (извлечение пользователя из cookie), requireAuth (требование авторизации), requireAdmin (требование роли администратора), requirePsychologist (требование роли психолога), requireBowls (требование роли специалиста по чашам или администратора).

5. Реализация бизнес-модулей — последовательная разработка 19 маршрутных модулей в директории server/routes/: аутентификация, пользователи, товары, корзина, курсы, расписание, записи, тибетские чаши, диагностика, трансляции, лаборатории, настройки, инструкция, медиа.

6. Реализация сигнального сервера — эндпоинты для обмена WebRTC-сигналами: POST для отправки сигнала, GET с параметром lastId для получения новых сигналов. Реализовано для двух типов комнат: видеоконференции (room.js) и трансляции (stream-room.js).

7. Реализация потоковой раздачи медиа — обработчик GET /uploads/:filename с определением MIME-типа по расширению, проверкой заголовка Range и формированием ответа 206 Partial Content или 200 OK.

8. Реализация модуля активности — функция logActivity() для серверного логирования; интеграция вызовов logActivity в маршруты авторизации, регистрации, записи на приём и бронирования.

9. Интеграция и тестирование — регистрация всех маршрутов в index.js, проверка TypeScript-совместимости с клиентской частью (npx tsc --noEmit), функциональное тестирование эндпоинтов.

4.2 Пользовательский интерфейс

Серверная часть не имеет собственного графического пользовательского интерфейса. Взаимодействие осуществляется через REST API. Описание пользовательского интерфейса клиентской части (React SPA) приведено в отдельной пояснительной записке по фронтенду.

4.3 Входные, выходные и промежуточные данные

Входные данные:
— HTTP-запросы от клиентской части с телом в формате JSON (например, {login: "user", password: "pass123"});
— параметры URL (path parameters): :id, :roomId, :streamId, :filename;
— параметры строки запроса (query parameters): ?city=biysk, ?search=Иванов, ?category=cosmetics, ?limit=50, ?lastId=10;
— файлы, загружаемые через multipart/form-data (изображения PNG/JPG, видео MP4/WebM, аудио MP3/WAV);
— cookies: поле token, содержащее UUID-токен аутентификации.

Промежуточные данные:
— записи в таблице tokens — активные сессии пользователей (создаются при входе, удаляются при выходе);
— записи в таблицах room_signals и stream_signals — WebRTC-сигналы, имеющие временный характер (очищаются при начале нового сеанса);
— файлы в директории /uploads — загруженные медиафайлы, пути к которым сохраняются в БД.

Выходные данные:
— JSON-ответы сервера с данными (объекты, массивы) или сообщениями об ошибках ({message: "Текст ошибки"});
— потоковые данные (видео, аудио) через HTTP Range с заголовками Content-Range и Content-Type;
— cookies: установка (при авторизации) или удаление (при выходе) токена аутентификации.

4.4.5 Алгоритмы использования технологий передачи данных

Алгоритм аутентификации (Cookie-Token):
1. Клиент отправляет POST /auth с телом {login, password}.
2. Сервер находит пользователя по логину в таблице users.
3. Если пользователь не найден — возвращается 400 «пользователь не найден».
4. Сервер вычисляет MD5(password) и сравнивает с полем password в БД.
5. Если хеши не совпадают — возвращается 400 «пароль неверный».
6. Сервер генерирует UUID-токен: token = crypto.randomUUID().
7. Сервер сохраняет токен в таблице tokens: INSERT INTO tokens (userId, token) VALUES (?, ?).
8. Сервер устанавливает cookie «token» с параметрами:
   — maxAge: 86400000 (24 часа в миллисекундах);
   — httpOnly: true (недоступен из клиентского JavaScript);
   — sameSite: 'none' (для кросс-доменных запросов);
   — secure: true (только HTTPS, в production).
9. Сервер вызывает logActivity(userId, login, fullName, 'Вход в систему', '').
10. Возвращается 200 {ok: true}.

Алгоритм проверки авторизации (middleware):
1. Из объекта req.cookies извлекается значение cookie «token».
2. Если cookie отсутствует — req.user = null, вызывается next().
3. По токену выполняется запрос: SELECT userId FROM tokens WHERE token = ?.
4. Если запись найдена — загружается пользователь: SELECT * FROM users WHERE id = ?.
5. Объект пользователя присваивается req.user.
6. Для защищённых эндпоинтов: requireAuth проверяет req.user !== null (иначе 401); requireAdmin проверяет req.user.isAdmin === 1 (иначе 403); requireBowls проверяет isBowlsSpecialist === 1 ИЛИ isAdmin === 1 (иначе 403).

Алгоритм WebRTC-сигнализации (видеоконференция):
1. Психолог инициирует сеанс: POST /appointments/:id/start.
2. Сервер обновляет статус записи: UPDATE appointments SET status = 'in_progress'.
3. Сервер очищает старые сигналы: DELETE FROM room_signals WHERE roomId = ?.
4. Инициатор создаёт RTCPeerConnection и локальный медиапоток (камера + микрофон).
5. Инициатор создаёт SDP offer через pc.createOffer() и устанавливает его как localDescription.
6. Инициатор отправляет POST /room/:roomId/signal с {type: 'offer', data: JSON.stringify(sdp)}.
7. Сервер выполняет INSERT INTO room_signals (roomId, senderId, type, data).
8. Второй участник опрашивает GET /room/:roomId/signals?lastId=N каждые 1-2 секунды.
9. Сервер возвращает SELECT * FROM room_signals WHERE roomId = ? AND id > ? AND senderId != ?.
10. Участник получает offer, создаёт answer через pc.createAnswer() и отправляет обратно.
11. Оба участника обмениваются ICE-кандидатами ({type: 'candidate', data: candidate}).
12. После обмена всеми сигналами устанавливается прямое P2P-соединение между браузерами.

Алгоритм потоковой раздачи медиа (HTTP Range):
1. Клиент запрашивает GET /uploads/:filename.
2. Сервер проверяет существование файла: fs.existsSync(filePath).
3. Если файл не найден — возвращается 404.
4. Сервер определяет размер файла: fs.statSync(filePath).size.
5. Сервер определяет MIME-тип по расширению файла из таблицы соответствий (.mp4 -> video/mp4, .mp3 -> audio/mpeg, .jpg -> image/jpeg и т.д.).
6. Проверяется наличие заголовка Range в запросе.
7. Если Range присутствует (например, «bytes=1048576-2097151»):
   a. Парсятся значения start и end.
   b. Создаётся ReadStream: fs.createReadStream(filePath, {start, end}).
   c. Формируется ответ 206 с заголовками:
      Content-Range: bytes start-end/totalSize
      Content-Length: end - start + 1
      Content-Type: определённый MIME-тип
   d. Поток направляется в ответ: stream.pipe(res).
8. Если Range отсутствует:
   a. Формируется ответ 200 с заголовками Content-Length и Accept-Ranges: bytes.
   b. Полный файл передаётся через fs.createReadStream(filePath).pipe(res).

Алгоритм логирования активности:
1. В ключевых маршрутах (auth, user, appointments, bowls-schedule, diagnostics-schedule) после успешного выполнения основной операции вызывается функция logActivity(userId, login, fullName, action, details).
2. Функция выполняет INSERT INTO user_activity_log (userId, userLogin, userFullName, action, details) VALUES (?, ?, ?, ?, ?).
3. Вызов обёрнут в try-catch для предотвращения прерывания основного потока при сбоях логирования.
4. Типы логируемых действий: «Вход в систему», «Регистрация», «Запись к психологу», «Запись на тибетские чаши», «Запись на диагностику».

4.5.1 Структурная организация программной системы

Система «Harmony Spa» построена по двухзвенной клиент-серверной архитектуре:

[Клиент: React SPA, порт 3000] <--HTTP/REST--> [Сервер: Express.js, порт 3001] <--SQL--> [SQLite: database.db]

Клиентская часть (React SPA) взаимодействует с серверной частью исключительно через HTTP REST API. Серверная часть развёрнута как единый Node.js процесс, обслуживающий все API-запросы и выполняющий роль:
— бизнес-логического сервера (обработка запросов, валидация, авторизация);
— сервера данных (CRUD-операции с SQLite);
— сигнального сервера (обмен WebRTC-метаданными);
— файлового сервера (потоковая раздача медиа).

5 ТЕСТИРОВАНИЕ И ОПТИМИЗАЦИЯ

5.1 План тестирования

Для серверной части системы «Harmony Spa» выбраны следующие виды тестирования:

1. Функциональное тестирование API — проверка корректности работы каждого эндпоинта путём отправки HTTP-запросов с различными входными данными и проверки кодов ответов, формата и содержания ответов. Данный вид тестирования является наиболее важным для REST API сервера, так как охватывает все функциональные требования.

2. Тестирование безопасности — проверка корректности разграничения доступа: попытки обращения к защищённым эндпоинтам без авторизации (ожидается 401), с недостаточными правами (ожидается 403).

3. Интеграционное тестирование — проверка взаимодействия серверной части с клиентской: TypeScript-компиляция клиента (npx tsc --noEmit) для подтверждения корректности типизации API-ответов.

Обоснование отсутствия других видов тестирования:
— Нагрузочное тестирование не проводится, так как система рассчитана на небольшое число одновременных пользователей (до нескольких десятков).
— UI-тестирование не применимо к серверной части, не имеющей графического интерфейса.
— Юнит-тестирование отдельных функций заменено функциональным тестированием эндпоинтов, так как бизнес-логика сосредоточена в маршрутных обработчиках.

5.2 Результаты тестирования и оптимизация

Все 12 тестов из плана тестирования пройдены успешно. Дополнительно выполнены следующие проверки:

— TypeScript-компиляция клиентской части (npx tsc --noEmit в директории client/) завершилась без ошибок, что подтверждает корректность типизации API-ответов и совместимость серверного API с клиентским кодом.

— Проверка загрузки серверных модулей (node -e "require('./db/activity'); require('./routes/activity')") выполнена без ошибок, что подтверждает корректность импортов и зависимостей.

— Проверка идемпотентности запуска: многократный перезапуск сервера не приводит к ошибкам, миграции корректно обрабатывают как новую, так и существующую базу данных.

Выполненные оптимизации:

1. Идемпотентные миграции — все ALTER TABLE обёрнуты в try-catch, что позволяет серверу корректно запускаться независимо от текущего состояния БД. Миграция применяется только если колонка ещё не существует.

2. Минимизация дублирования сигналов — в алгоритме long polling для WebRTC используется параметр lastId, благодаря которому клиент получает только новые сигналы с id > lastId, избегая повторной обработки.

3. Потоковая раздача через HTTP Range — вместо загрузки полного файла в память используется fs.createReadStream с параметрами start и end, что обеспечивает минимальное потребление памяти при раздаче больших медиафайлов.

4. Денормализация в журнале активности — логин и ФИО пользователя сохраняются непосредственно в записи журнала, что позволяет выполнять поиск по журналу одним SQL-запросом без JOIN с таблицей users.


6 РУКОВОДСТВО ПОЛЬЗОВАТЕЛЯ

Серверная часть не имеет графического интерфейса. Руководство описывает порядок установки, настройки и запуска сервера.

Требования к окружению:
— Node.js версии 18 или выше (проверка: node --version);
— npm (устанавливается вместе с Node.js);
— операционная система: Windows 10+, Linux или macOS.

Порядок установки и запуска:

1. Перейти в директорию серверной части:
   cd server

2. Установить зависимости:
   npm install

3. Запустить сервер:
   node index.js

4. Убедиться, что в консоли появилось сообщение:
   «Example app listening on port 3001!»

5. Сервер доступен по адресу: http://localhost:3001

При первом запуске автоматически выполняются следующие действия:
— создаётся файл базы данных database.db;
— создаются все 22 таблицы;
— создаётся директория uploads/ (если отсутствует);
— создаются тестовые учётные записи:
  * admin / admin123 — администратор;
  * psyholog / psyhopog123 — психолог (Коюшева Оксана Викторовна);
  * bowls / bowls123 — специалист по тибетским чашам;
  * user / user123 — обычный пользователь;
— заполняется инструкция (10 пунктов навигации по сайту);
— создаётся тестовая трансляция «Утренняя медитация и дыхательные практики».

Действия в случае сбоя:
— Если сервер не запускается с ошибкой «Cannot find module» — повторить npm install.
— Если порт 3001 занят — задать переменную окружения: PORT=3002 node index.js.
— Для сброса данных — удалить файл database.db и перезапустить сервер.

Проверка работоспособности:
— Открыть в браузере http://localhost:3001 — должен вернуться JSON {ok: true}.
— Отправить GET http://localhost:3001/products — должен вернуться массив товаров.


ЗАКЛЮЧЕНИЕ

В результате выполнения выпускной квалификационной работы разработана серверная часть веб-приложения «Harmony Spa» для центра здоровья и красоты. Поставленная задача решена полностью. Выполнены следующие работы:

1. Проведён анализ предметной области, выявлены 18 информационных объектов и определены связи между ними. Рассмотрены 4 аналога (Yclients, DIKIDI, Bookly, Zoom), выполнен сравнительный анализ, подтвердивший высокую степень новизны разрабатываемого решения.

2. Спроектирована и реализована реляционная база данных на SQLite, содержащая 22 таблицы с внешними ключами и ограничениями целостности. Реализован механизм идемпотентных миграций.

3. Реализован REST API на Node.js + Express.js, состоящий из 19 маршрутных модулей и 9 модулей доступа к данным. API покрывает 42 функциональных требования.

4. Разработана система аутентификации на основе UUID-токенов в HTTP-only cookies с ролевым разграничением доступа через middleware (5 ролей).

5. Реализован сигнальный сервер для WebRTC, обеспечивающий видеоконференцсвязь (приём психолога, диагностика) и прямые трансляции с чатом.

6. Реализована потоковая раздача медиафайлов с поддержкой HTTP Range для видео и аудио контента.

7. Реализована система журналирования активности пользователей с серверным логированием ключевых действий.

8. Проведено функциональное тестирование (12 тестов) и интеграционное тестирование с клиентской частью (TypeScript-компиляция). Все тесты пройдены успешно.

Объём серверного кода: index.js (105 строк) + 9 модулей доступа к данным + 19 маршрутных модулей. Ключевой модуль db.js — 426 строк. Общий объём серверной части — около 3000 строк кода JavaScript.

Возможные точки роста (развития) системы:
— переход на PostgreSQL при увеличении нагрузки и числа одновременных пользователей;
— замена long polling на WebSocket для уменьшения задержки при обмене WebRTC-сигналами;
— внедрение HTTPS для повышения безопасности в production;
— расширение аналитики — добавление статистических отчётов по активности пользователей;
— добавление механизма push-уведомлений о предстоящих записях.
