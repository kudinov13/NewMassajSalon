ПЕРЕЧЕНЬ СОКРАЩЕНИЙ, УСЛОВНЫХ ОБОЗНАЧЕНИЙ И ТЕРМИНОВ

API (Application Programming Interface) — интерфейс прикладного программирования, набор правил и спецификаций, позволяющих программным компонентам взаимодействовать друг с другом.
REST (Representational State Transfer) — архитектурный стиль взаимодействия компонентов распределённого приложения в сети, основанный на использовании стандартных методов протокола HTTP.
HTTP (HyperText Transfer Protocol) — протокол прикладного уровня для передачи гипертекстовых документов.
CRUD (Create, Read, Update, Delete) — четыре базовые операции работы с данными: создание, чтение, обновление и удаление.
SQLite — компактная встраиваемая реляционная СУБД, не требующая отдельного серверного процесса и хранящая базу данных в одном файле.
SQL (Structured Query Language) — язык структурированных запросов, предназначенный для управления данными в реляционных СУБД.
Node.js — серверная среда выполнения JavaScript на основе движка V8.
Express.js — минималистичный веб-фреймворк для Node.js, предоставляющий средства для создания REST API.
WebRTC (Web Real-Time Communication) — технология передачи потоковых аудио- и видеоданных между браузерами в реальном времени.
SDP (Session Description Protocol) — протокол описания параметров мультимедийной сессии в WebRTC.
ICE (Interactive Connectivity Establishment) — протокол установления соединения в WebRTC.
Middleware — промежуточное программное обеспечение, функция, выполняемая в цепочке обработки HTTP-запроса на сервере.
SPA (Single Page Application) — одностраничное приложение, взаимодействующее с сервером через API.
CORS (Cross-Origin Resource Sharing) — механизм, позволяющий веб-страницам выполнять запросы к ресурсам на другом домене.
Long Polling — техника периодического опроса сервера клиентом для получения новых данных.
Эндпоинт (endpoint) — конкретный URL-адрес на сервере, обрабатывающий определённый тип запроса.
Сигнальный сервер — серверный компонент, обеспечивающий обмен метаданными между участниками WebRTC-соединения.

ВВЕДЕНИЕ
============================================================

В современном обществе наблюдается рост интереса к альтернативным методам оздоровления, психологической поддержке и комплексному подходу к здоровью. Центры здоровья и красоты нуждаются в автоматизации бизнес-процессов для повышения эффективности работы.

Актуальность работы обусловлена тем, что большинство подобных центров ведут запись вручную, не имеют единой системы учёта, а консультации проводят через сторонние сервисы (Zoom, Skype), не интегрированные с системой записи.

Существующие решения (Yclients, DIKIDI, Bookly) ориентированы на типовые салоны и не поддерживают специфику: видеоконсультации по диагностике, запись на массаж тибетскими чашами с выбором города, продажу видеокурсов, прямые трансляции и систему отслеживания активности.

Целью ВКР является разработка серверной части веб-приложения «Harmony Spa», обеспечивающей: управление БД, аутентификацию, ролевую модель доступа, REST API для всех модулей, сигнальный сервер для видеоконференций и потоковую раздачу медиафайлов.

Задачи:
- проектирование реляционной БД (22 таблицы) на SQLite;
- реализация REST API на Node.js + Express.js (19 маршрутных модулей);
- система аутентификации на токенах через HTTP-only cookies;
- сигнальный сервер WebRTC для видеоконференций и трансляций;
- потоковая раздача видео/аудио с поддержкой HTTP Range;
- тестирование.

1 ПОСТАНОВКА ЗАДАЧИ ВКР
============================================================

1.1 Бизнес-требования

Заказчик — центр здоровья «Harmony Spa» (города Новосибирск и Бийск). Бизнес-цели:
1. Автоматизация записи клиентов на приём к психологу, тибетский массаж и диагностику.
2. Онлайн-консультации — встроенные видеоконференции без сторонних сервисов.
3. Электронная коммерция — продажа товаров и видеокурсов.
4. Прямые трансляции мастер-классов с чатом.
5. Мониторинг активности пользователей.
6. Мультигородская работа — раздельное расписание для Бийска и Новосибирска.

1.2 Пользовательские требования

Для клиентов: регистрация/авторизация, каталог, корзина, запись к психологу, запись на чаши (с выбором города), запись на диагностику, видеоконференции, курсы, трансляции, история покупок.

Для психолога: управление расписанием, записи клиентов, видеоконференции, поиск пользователей, журнал активности.

Для специалиста по чашам: управление расписанием с городом, записи клиентов, загрузка медиа, поиск пользователей.

Для администратора: полный доступ ко всему: товары, курсы, трансляции, пользователи, диагностика, прямые эфиры, журнал активности.

1.3 Системные требования

Система — REST API сервер в клиент-серверной архитектуре. Подсистемы:
1. Аутентификация и авторизация
2. Каталог товаров
3. Корзина и заказы
4. Курсы
5. Расписание и записи (психолог, чаши, диагностика)
6. Видеоконференцсвязь (WebRTC сигнализация)
7. Трансляции (эфиры, чат)
8. Медиа-контент
9. Журнал активности
10. Настройки и контент

Требования к среде: Node.js 18+, любая ОС, 512 МБ RAM.

1.4 Требования к пользовательскому интерфейсу

Серверная часть не имеет GUI. Взаимодействие через REST API:
- Ответы в JSON, коды HTTP (200, 400, 401, 403, 404).
- Ошибки на русском в поле «message».
- Аутентификация через HTTP-only cookies.
- CORS для работы с клиентом на другом порту.

2 АНАЛИЗ ТРЕБОВАНИЙ И ОПРЕДЕЛЕНИЕ СПЕЦИФИКАЦИЙ
============================================================

2.1 Описание предметной области

Предметная область — автоматизация многопрофильного центра здоровья: психологическое консультирование, тибетский массаж (уникальная процедура «в 4 руки»), визуальная диагностика, продажа товаров и курсов, трансляции.

Центр работает в двух городах, приём психолога и диагностика — дистанционно по видео. Ключевые проблемы: отсутствие единой системы записи, необходимость встроенной видеосвязи, потребность в мониторинге.

2.1.1 Информационные объекты и взаимосвязи

Основные объекты: Пользователь, Товар, Корзина, Заказ, Курс, Видеоурок, Слот расписания, Запись на приём, Комната видеосвязи, Трансляция, Журнал активности, Настройки, Лаборатория, Медиа чаш, Инструкция.

Связи: Пользователь → Корзина, Заказы, Записи, Курсы, Трансляции, Активность. Слоты → Записи. Курсы → Видеоуроки. Товары → Курсы (опционально).

2.1.2 Информационные потребности пользователей

Клиенты: каталог, расписание, статусы записей, курсы, история.
Специалисты: записи клиентов с контактами, расписание, поиск пользователей, журнал.
Администратор: статистика, журнал, управление ролями и контентом.

2.1.3 Методы работы с информационными объектами

2.1.3.1 Способы хранения: SQLite (реляционная БД в одном файле, нулевая конфигурация, поддержка SQL и FK). Медиафайлы — в файловой системе /uploads.

2.1.3.2 Математические модели: хеширование MD5, генерация UUID, правило «2 часа до приёма», агрегация SQL.

2.1.3.5 Технологии передачи: REST API (JSON через HTTP), HTTP-only Cookies, HTTP Range (потоковое видео), Long Polling (WebRTC-сигналы), WebRTC сигнализация (SDP/ICE).

2.1.4 Обзор аналогов

Yclients — нет видеосвязи, нет мультигородского расписания, платная (от 1500 р/мес).
DIKIDI — нет видео, нет товаров/курсов, нет трансляций.
Bookly — привязка к WordPress, нет видео, нет e-commerce.
Zoom/Meet — не интегрированы с записью, требуют регистрации.

Вывод: ни одно решение не объединяет все требуемые функции.

2.1.5 Концептуальное обоснование

Оригинальность: интеграция WebRTC-сигнализации в основной сервер; единая БД для всех модулей; совмещение ролей; мультигородское расписание через параметрическую фильтрацию.

2.2 Классы пользователей

1. Гость — просмотр каталога, описаний.
2. Клиент (isAdmin=0, isPsychologist=0, isBowlsSpecialist=0) — покупки, записи, видео.
3. Психолог (isPsychologist=1) — расписание, записи клиентов, видеоконференции, поиск.
4. Специалист по чашам (isBowlsSpecialist=1) — расписание с городом, записи, медиа.
5. Администратор (isAdmin=1) — полный доступ.

Роли не взаимоисключающие.

2.3 Функциональные требования

2.3.1 Функциональные возможности (42 функции по модулям):

Аутентификация: F1-F5 (регистрация, вход, выход, профиль, обновление).
Каталог: F6-F8 (список, CRUD, загрузка изображений).
Корзина: F9-F12 (добавление, просмотр, оформление, история).
Курсы: F13-F15 (создание, покупка, просмотр).
Расписание психолога: F16-F19 (слоты, бронирование, видеосеанс).
Тибетские чаши: F20-F23 (слоты с городом, бронирование, записи).
Диагностика: F24-F26 (слоты, бронирование, активация комнаты).
Видеоконференции: F27-F28 (сигналы WebRTC).
Трансляции: F29-F33 (CRUD, покупка, эфир, чат, превью).
Медиа: F34-F35 (загрузка, потоковая раздача).
Активность: F36-F38 (логирование, поиск по журналу, поиск пользователей).
Настройки: F39-F42 (настройки, инструкция, лаборатории, роли).

2.3.2 Описание прецедентов

Прецедент 1 — Регистрация: POST /user → проверка уникальности → хеш пароля → создание → лог активности → ответ 200.

Прецедент 2 — Бронирование психолога: POST /appointments → проверка слота → правило 2 часов → генерация roomId → пометка слота → создание записи → лог → ответ.

Прецедент 3 — Видеоконференция: POST start → очистка сигналов → обмен через POST/GET signals (offer→answer→ICE) → P2P → POST complete.

Прецедент 4 — Запись на чаши: GET /bowls-schedule?city=X → POST /bowls-schedule/book → проверка → бронирование → лог.

Прецедент 5 — Прямая трансляция: POST go-live → генерация streamRoomId → вещание через WebRTC → зрители подключаются → чат.

2.4 Нефункциональные требования

Производительность: ответ <200мс, SQLite до 100K INSERT/сек.
Совместимость: Node.js 18+ на Windows/Linux/macOS.
Надёжность: идемпотентные миграции (try-catch), автосоздание таблиц.
Безопасность: MD5-хеши, httpOnly cookies, ролевой middleware.
Развёртывание: npm install && node index.js.

3.1 Сравнительный анализ

Серверная среда: Node.js выбран за единство языка с фронтом, нативную асинхронность, npm-экосистему (2M+ пакетов). Альтернативы (Python/Django, PHP/Laravel, Java/Spring) требуют другого языка.

Фреймворк: Express.js выбран как стандарт Node.js с крупнейшим сообществом. Альтернативы (Fastify, Koa, NestJS) менее зрелы или избыточны.

СУБД: SQLite выбрана за нулевую конфигурацию и хранение в файле. PostgreSQL/MySQL избыточны для данной нагрузки. MongoDB не поддерживает FK.

3.2 Характеристика выбранных средств

- Node.js v18+ — серверный JavaScript, событийная модель, неблокирующий I/O.
- Express.js v4 — маршрутизация, middleware, статика.
- sqlite3 + sqlite — привязка к C++ SQLite, Promise/async-await.
- md5 — хеширование паролей.
- cookie-parser — парсинг cookies.
- cors — CORS-заголовки.
- multer — загрузка файлов (multipart/form-data).
- crypto (встроенный) — генерация UUID-токенов.
- fs, path (встроенные) — работа с файлами.
- VS Code — среда разработки.

Входные: HTTP-запросы (JSON body, path/query params, files, cookies).
Промежуточные: записи в tokens, room_signals, stream_signals; файлы в /uploads.
Выходные: JSON-ответы, потоковые данные, cookies.

4.4 Реализация методов хранения и передачи

4.4.1 Методы хранения данных — БД SQLite, 22 таблицы:

products (id, name, description, price, oldPrice, category, image, courseId, partnerUrl, createdAt)
cart (id, userId, productId, quantity) UNIQUE(userId, productId)
orders (id, userId, total, createdAt)
order_items (id, orderId, productId, name, price, quantity)

4.4.5 Алгоритмы передачи данных

Аутентификация: POST /auth → MD5(password) → сравнение → UUID токен → cookie (httpOnly, maxAge 24h).

WebRTC-сигнализация: Инициатор → POST signal (offer) → Сервер хранит → Второй GET signals (lastId) → answer → ICE → P2P.

HTTP Range: GET /uploads/file → проверка Range → fs.createReadStream({start, end}) → 206 Partial Content.

Ролевая авторизация: cookie → tokens → userId → users → проверка роли → 401/403 или next().
